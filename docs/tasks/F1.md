# F1 — API core: especificaciones

> **Objetivo de la fase**: la API (`apps/api`) levantada con auth por email completa y multi-tenancy funcionando. Al cerrar F1, cualquier app puede autenticar usuarios y hacer scoping por organización.
> **Gate de salida**: suite de aislamiento (F1-09) verde sobre todos los endpoints existentes · API deployada respondiendo `/healthz` · flujo register→verify→login→refresh ejecutado contra producción.
>
> DoD global: ver [F0.md](F0.md). Además, en F1: **todo endpoint nuevo queda registrado en la tabla de políticas de F1-09** desde el PR que lo crea (una vez que F1-09 exista).

## F1-01 ⛔ — Scaffolding de la API (`apps/api`)

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F0-01, F0-02 · **Bloquea**: todo F1
**Docs**: [Técnico §3, §6](../03-tecnico.md) · [Arquitectura §9](../02-arquitectura.md) · v1: `bv-cross/server/src/app.ts` (patrones a portar)

### Objetivo
Esqueleto de la API con middlewares de seguridad, config validada, manejo de errores y observabilidad mínima — sin ningún módulo de negocio todavía.

### Alcance
**Incluye**: bootstrap, `app.ts`, `config.ts`, `lib/errors.ts`, `lib/http.ts`, logger, `/healthz`, Dockerfile.
**No incluye**: conexión Mongo (F1-02), auth (F1-04+), rate limiting (F1-06).

### Especificación técnica

**`config.ts`** — schema Zod, `process.exit(1)` con mensaje claro si falla:

| Var | Tipo/validación | Default |
|---|---|---|
| `NODE_ENV` | `development \| test \| production` | `development` |
| `PORT` | int 1..65535 | `8787` |
| `MONGODB_URI` | url `mongodb(+srv)://` | requerida |
| `JWT_SECRET` | string min 32 chars | requerida (en dev puede ir en `.env`) |
| `ACCESS_TOKEN_TTL_MIN` | int 5..60 | `15` |
| `REFRESH_TOKEN_TTL_DAYS` | int 1..90 | `30` |
| `COOKIE_DOMAIN` | string que empieza con `.` | vacío en dev (host-only) |
| `APP_ORIGINS` | CSV de URLs https | vacío en dev |
| `EMAIL_FROM` | email | `no-reply@localhost` |
| `RESEND_API_KEY` | string | opcional (sin ella → provider consola) |
| `TRUST_PROXY` | bool | `true` si prod |

**`app.ts`** — orden de middlewares (importa: cada uno asume el anterior):
1. `requestId` (crypto.randomUUID → contexto + header `X-Request-Id`).
2. Logger **pino** (no el `logger()` de Hono): JSON con `requestId`, método, ruta, status, `durationMs`; `/healthz` excluido.
3. `secureHeaders` (portar config CSP de v1; en API pura la CSP es mínima: `default-src 'none'`).
4. **CORS**: allowlist exacta desde `APP_ORIGINS` + `credentials: true` + métodos `GET,POST,PATCH,DELETE` + headers `Authorization, Content-Type, X-Org-Id`. En dev: permitir `localhost:517*`.
5. Guard de `Origin` en mutaciones (portar de v1 — defensa CSRF adicional; responde `BAD_ORIGIN` 403).
6. `bodyLimit` 16 KB.
7. `Cache-Control: private, no-store` en `/api/*` (patrón v1).
8. Rutas → 404 `NOT_FOUND` para `/api/*` desconocido → `onError`.

**`lib/errors.ts`**: `DomainError` según [Técnico §6](../03-tecnico.md); `onError` mapea `DomainError` → envelope con status del catálogo de `@bv/contracts`; cualquier otro error → log con stack + `INTERNAL` 500 **sin detalles al cliente**. `ZodError` en parseo de rutas → `VALIDATION_ERROR` 400 con `details` = issues aplanadas.

**`lib/http.ts`**: helpers `ok(c, data, status?)`, `parseBody(c, schema)`, `parseQuery(c, schema)` (lanzan `DomainError('VALIDATION_ERROR')`).

**`/healthz`**: `{ status: 'ok', uptime, version }` (agrega `mongo: ok|down` cuando exista F1-02). Sin datos sensibles.

**Dockerfile** (multi-stage, contexto = raíz del monorepo):
`base` (node:22-slim + pnpm) → `pruner` (`turbo prune @bv/api --docker`) → `builder` (install frozen + build) → `runner` (solo `dist` + prod deps, user no-root, `NODE_ENV=production`).

### Casos de prueba
1. `GET /healthz` → 200 `{status:'ok'}`.
2. `GET /api/v1/nope` → 404 envelope `{error:{code:'NOT_FOUND'}}`.
3. Body de 17 KB a un endpoint POST cualquiera → 413.
4. Preflight OPTIONS desde origen permitido → headers CORS correctos; desde origen desconocido → sin `Access-Control-Allow-Origin`.
5. POST con header `Origin` distinto de self/allowlist → 403 `BAD_ORIGIN`.
6. Arranque sin `JWT_SECRET` → exit code 1 y mensaje que nombra la variable (test de proceso hijo).
7. Error inesperado lanzado en una ruta de prueba → 500 `INTERNAL`, sin stack en el body, stack presente en el log.
8. `docker build` desde la raíz termina y la imagen arranca (`docker run … /healthz` 200) — smoke manual documentado.

### Criterios de aceptación
- [ ] `pnpm --filter @bv/api dev` levanta con `.env` de ejemplo; `.env.example` completo commiteado.
- [ ] Los 8 casos de prueba verdes (los manuales, evidenciados en el PR).
- [ ] Log de una request muestra `requestId` y `durationMs` en JSON.
- [ ] Ningún `console.log` — todo por pino.

## F1-02 ⛔ — Capa Mongo

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F1-01 · **Bloquea**: todo módulo
**Docs**: [Arquitectura §4](../02-arquitectura.md) (modelo completo) · [Testing §1](../08-testing.md) · DEC-02/03

### Objetivo
Conexión Mongo tipada, todos los índices declarados y aplicados al boot, y el harness de tests de integración con transacciones.

### Especificación técnica

**`db/client.ts`**: `MongoClient` singleton — opciones: `serverSelectionTimeoutMS: 5000`, `maxPoolSize: 20`, `retryWrites: true`, `w: 'majority'`. `connect()` con ping al boot (falla → exit 1). Cierre limpio en `SIGTERM`/`SIGINT` (drena y `client.close()`).

**`db/collections.ts`**: getters tipados con interfaces de documento **internas** (`UserDoc`, `MembershipDoc`, …) — los tipos de documento NO viven en `@bv/contracts` (ahí van DTOs; el doc interno tiene `passwordHash` etc.).

**`db/indexes.ts`**: array `IndexSpec[]` con **todos** los índices de [Arquitectura §4](../02-arquitectura.md), transcriptos 1:1 (users.email unique; organizations.joinCode/slug unique; memberships `{orgId,userId}` unique + 3 más; refreshTokens tokenHash unique + TTL `expiresAt`; exercises ×2; rmEntries ×2; classTemplates; classSessions `{orgId,startsAt}` + `{templateId,startsAt}` unique; packs; packAssignments ×3; bookings unique **parcial** `{sessionId,userId}` con `partialFilterExpression: {status:'booked'}` + 2 más; emailTokens tokenHash unique + TTL). `ensureIndexes()` idempotente al boot; loguea creados vs existentes.

**Harness de tests** (`test/setup.ts`): `mongodb-memory-server` con `MongoMemoryReplSet` (1 nodo — habilita transacciones); DB con nombre aleatorio por suite; helper `testDb()` que conecta, corre `ensureIndexes()` y expone `cleanup()` (drop DB). Se registra como setup global de Vitest para el workspace api.

### Casos de prueba
1. `ensureIndexes()` dos veces seguidas → sin error, segunda corrida no crea nada.
2. Insertar 2 users mismo email (case distinto: el repo guarda lowercase) → E11000.
3. Insertar 2 bookings `{sessionId,userId}` ambos `status:'booked'` → E11000; segundo con `status:'cancelled_by_user'` → permitido (índice parcial funciona).
4. Documento en `refreshTokens` con `expiresAt` pasado → TTL lo elimina (en test: verificar que el índice tiene `expireAfterSeconds: 0`, no esperar al monitor).
5. Transacción de prueba (insertar en 2 colecciones, abortar) → ninguna escritura visible.
6. `SIGTERM` → proceso cierra la conexión (test de humo con child process, o unitario del handler).

### Criterios de aceptación
- [ ] Índices en código = índices en [Arquitectura §4](../02-arquitectura.md), revisión 1:1 en el PR (tabla comparativa en la descripción).
- [ ] Suite de integración corre local y en CI < 60 s.
- [ ] `/healthz` reporta `mongo: 'ok'` y pasa a `down` si se corta la conexión (test manual documentado).

## F1-03 ⛔ — `lib/crypto.ts` y `lib/email.ts`

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F1-01 · **Bloquea**: F1-04
**Docs**: [Seguridad §1](../05-seguridad.md) · v1: `bv-cross/server/src/lib/crypto.ts`

### Especificación técnica

**Crypto** (portar de v1 — los hashes v1 migrados deben seguir verificando):
- `hashPassword(plain)` / `verifyPassword(plain, stored)`: scrypt `N=2^15, r=8, p=1` para hashes **nuevos**, salt 16 bytes, formato almacenado `scrypt$N$r$p$salt_b64$hash_b64`; comparación con `timingSafeEqual`. `verifyPassword` **parsea N/r/p del hash almacenado** (con cota superior anti-DoS): v1 hasheaba con `N=2^14` y su verify usaba constantes, así que la paridad exacta de parámetros no alcanza — el formato autodescriptivo garantiza la migración y permite subir parámetros a futuro.
- `fakeVerify()`: corre un verify contra un hash dummy fijo — se llama cuando el usuario no existe para uniformar timing.
- `generateToken()`: `randomBytes(32)` → base64url (43 chars). `hashToken(token)`: SHA-256 hex — lo único que se persiste.
- Passwords: helper `isWeakPassword(plain)` — `length < 8` o pertenece a lista embebida top-10k (archivo `data/common-passwords.txt`, comparación lowercase).

**Email** (`lib/email.ts`):
```ts
interface EmailProvider { send(input: { to: string; template: TemplateName; data: Record<string, string> }): Promise<void> }
```
- `ResendProvider` (fetch a API Resend, `from: EMAIL_FROM`; error → log + throw `DomainError('INTERNAL')`... **no**: el register no debe fallar si el mail falla → el servicio de auth decide; el provider solo lanza; auth captura, loguea y sigue — el usuario puede pedir reenvío).
- `ConsoleProvider` (dev/test): imprime destinatario + **link completo** al log.
- Selección por config: sin `RESEND_API_KEY` → consola.
- Templates (HTML simple + texto plano, español, sin tracking): `verify-email` (link `{{appUrl}}/verify?token=…`, expira 30 min), `reset-password` (30 min), `member-invite` (nombre del gym + joinCode + links a las apps).

### Casos de prueba
1. `verifyPassword` OK con hash recién creado; falla con password incorrecta; falla con hash adulterado (un char cambiado).
2. Hash de v1 real (fixture copiada de la DB v1) verifica con la password conocida — **garantía de migración**.
3. Dos `hashPassword` de la misma password → hashes distintos (salt).
4. `generateToken` → 100 tokens únicos, base64url estricto (sin `+/=`).
5. `isWeakPassword`: `"1234567"` true (corta), `"password"` true (lista), `"Xk9#mQ2v"` false.
6. `ConsoleProvider` loguea el link con el token; `ResendProvider` con API key inválida lanza (mock de fetch).
7. Templates renderizan sin `{{placeholders}}` sin resolver (test que busca `{{` en el output).

### Criterios de aceptación
- [ ] Timing uniforme estructural: el servicio de login (F1-04) puede llamar `fakeVerify` — exportado y testeado acá.
- [ ] En dev, el flujo completo es operable sin cuenta de Resend (link por consola).
- [ ] Ningún token ni password aparece en logs de nivel `info`+ (los links van por el provider consola, que solo corre en dev).

## F1-04 ⛔ — Módulo `auth`: register + verify + login

**Dificultad**: 🔴 Senior · **Modelo**: Fable · **Depende de**: F1-02, F1-03 · **Bloquea**: F1-05..12
**Docs**: [Arquitectura §5](../02-arquitectura.md) · [Seguridad §1](../05-seguridad.md) · RN-24 · DEC-04/05

### Objetivo
Registro con verificación por email y login que emite el par access/refresh según DEC-04.

### Alcance
**Incluye**: register, verify-email, resend-verification, login, emisión de tokens (servicio `token-service` reutilizable). **No incluye**: refresh/logout/reset (F1-05), rate limiting (F1-06 — pero los endpoints se diseñan para recibirlo).

### Especificación técnica

**Endpoints** (`modules/auth/auth.routes.ts`):

| Método y ruta | Body (schema en `@bv/contracts`) | Respuesta | Errores |
|---|---|---|---|
| `POST /api/v1/auth/register` | `{ email, password, name: 1..80 }` | 201 `{ user: UserDto }` | `EMAIL_TAKEN` 409 · `WEAK_PASSWORD` 400 |
| `POST /api/v1/auth/verify-email` | `{ token }` | 204 | `TOKEN_INVALID` 401 *(status del catálogo `@bv/contracts` — fuente única; antes decía 400)* |
| `POST /api/v1/auth/resend-verification` | `{ email }` | **202 siempre** | — (anti-enumeración) |
| `POST /api/v1/auth/login` | `{ email, password }` | 200 `{ accessToken, user: UserDto, memberships: MembershipSummaryDto[] }` + `Set-Cookie` refresh | `INVALID_CREDENTIALS` 401 · `EMAIL_NOT_VERIFIED` 403 |

**Reglas de implementación**:
- Email siempre normalizado por el schema (`trim().toLowerCase()`); unicidad por índice — la race de doble registro se resuelve capturando E11000 → `EMAIL_TAKEN`.
- Register: crea `users` (`emailVerifiedAt: null`) + `emailTokens` (`purpose:'verify'`, `expiresAt: now+30min`, se guarda `hashToken`) + envía email. Si el envío falla: **el registro NO falla** — log warn + el usuario usa resend.
- Verify: busca por `hashToken(token)`, valida `purpose`, `usedAt == null`, `expiresAt > now` → set `usedAt` + `users.emailVerifiedAt`. Cualquier fallo (inexistente/usado/expirado) → mismo `TOKEN_INVALID` (sin oráculo).
- Resend: si el email existe y no está verificado → invalida tokens verify previos (set `usedAt`) y emite uno nuevo. Siempre 202.
- Login: si el email no existe → `fakeVerify()` y `INVALID_CREDENTIALS` (mismo código y timing que password errónea). Password OK pero sin verificar → `EMAIL_NOT_VERIFIED`. OK → `token-service.issuePair(userId)`:
  - **access**: JWT HS256, claims `{ sub: userId, iat, exp: +ACCESS_TOKEN_TTL_MIN }` — sin roles ni orgs adentro (eso lo resuelve `tenantGuard` por request).
  - **refresh**: `generateToken()`; persiste `{ tokenHash, userId, familyId: nuevo ObjectId, expiresAt: +REFRESH_TOKEN_TTL_DAYS, userAgent, ip }`; cookie `refresh_token` con `HttpOnly; Secure (prod); SameSite=Lax; Domain=COOKIE_DOMAIN; Path=/api/v1/auth; Max-Age=…`.
- `memberships` en la respuesta de login: resumen para el selector de org (id, orgId, orgName, role, status) — solo `active` e `invited`.
- `UserDto`: `{ id, email, name, emailVerified: boolean }` — jamás hashes. `toDto()` explícito.

### Casos de prueba (integración, `app.request()`)
**Flujo feliz**: register 201 → email capturado (provider consola/mock) → verify 204 → login 200 con accessToken decodificable + cookie con todos los flags.
**Register**: email duplicado (y con casing distinto) → 409 · password débil → 400 con code `WEAK_PASSWORD` · body con campo extra → 400 (`.strict()`) · fallo del provider de email → 201 igual (mock que lanza).
**Verify**: token inexistente / ya usado / expirado (fake timers) → 401 `TOKEN_INVALID` en los tres.
**Resend**: email inexistente → 202 y **cero** emails enviados (spy) · email ya verificado → 202 y cero emails · reemite → el token viejo deja de servir.
**Login**: password errónea → 401 · email inexistente → 401 **y** `fakeVerify` invocado (spy — timing estructural) · sin verificar → 403 · verificado + 2 memberships sembradas → respuesta incluye ambas · cookie: `HttpOnly`, `Path=/api/v1/auth`, `SameSite=Lax` presentes (parse del header).
**Seguridad transversal**: en la DB no existe ningún token en claro (scan de la colección tras el flujo) · el log de la suite no contiene la password (búsqueda en output capturado).

### Criterios de aceptación
- [ ] Tabla de endpoints implementada exactamente (métodos, códigos, envelope).
- [ ] Todos los casos de prueba listados, verdes.
- [ ] Schemas agregados a `@bv/contracts` en este PR (`registerBody`, `loginBody`, `userDto`, `membershipSummaryDto`).
- [ ] `token-service` exportado como servicio (lo reusa F1-05) con unit tests propios de claims y expiración.

## F1-05 ⛔ — Módulo `auth`: refresh + logout + reset + change-password

**Dificultad**: 🔴 Senior · **Modelo**: Fable · **Depende de**: F1-04
**Docs**: [Seguridad §1](../05-seguridad.md) (rotación y detección de reuso) · DEC-04

### Objetivo
Completar el ciclo de vida de la sesión: renovación silenciosa, cierre, recuperación y cambio de contraseña — con detección de robo de refresh tokens.

### Especificación técnica

| Método y ruta | Entrada | Respuesta | Errores |
|---|---|---|---|
| `POST /auth/refresh` | cookie `refresh_token` | 200 `{ accessToken }` + nueva cookie | `TOKEN_INVALID` 401 |
| `POST /auth/logout` | cookie | 204 + cookie vaciada (`Max-Age=0`) | — (idempotente: sin cookie → 204) |
| `POST /auth/forgot-password` | `{ email }` | **202 siempre** | — |
| `POST /auth/reset-password` | `{ token, newPassword }` | 204 | `TOKEN_INVALID` 401 (catálogo) · `WEAK_PASSWORD` 400 |
| `POST /auth/change-password` | `{ currentPassword, newPassword }` [requireAuth] | 204 | `INVALID_CREDENTIALS` 401 · `WEAK_PASSWORD` 400 |

**Rotación con detección de reuso** (el corazón de la tarea):
1. `refresh` busca por `hashToken`. Casos:
   - No existe / `expiresAt < now` → 401.
   - **`revokedAt != null` → REUSO**: alguien presentó un token ya rotado. Respuesta 401 **y** revocación inmediata de TODA la familia (`updateMany({familyId}, {$set:{revokedAt}})`) + log `warn` con `userId`, `familyId`, ip/UA (evento de seguridad).
   - Válido → marca `revokedAt` en el actual e inserta uno nuevo **misma `familyId`** (rotación); emite access nuevo. Ambas escrituras en una transacción (el par marca+inserta debe ser atómico).
2. `logout`: revoca la familia de la cookie presentada.
3. `forgot-password`: si el email existe → invalida tokens `reset` previos y emite nuevo (30 min). Respuesta idéntica exista o no.
4. `reset-password`: valida token (mismas reglas que verify) → `hashPassword(new)` → **revoca TODAS las familias de refresh del usuario** + marca token usado. También marca `emailVerifiedAt` si era null (probó control del email).
5. `change-password`: verifica `currentPassword` → actualiza hash → revoca todas las familias **excepto** la de la sesión actual (UX: no desloguear al que cambió).

### Casos de prueba
**Rotación**: login → refresh → access nuevo distinto, cookie nueva distinta; el refresh viejo reintentado → 401 **y** el nuevo también queda revocado (familia caída) — el escenario de robo completo.
**Cadena**: 5 refresh seguidos → 5 documentos, familia única, solo el último sin `revokedAt`.
**Logout**: tras logout, refresh → 401; logout sin cookie → 204.
**Forgot/reset**: email inexistente → 202 y cero emails (spy) · reset con token OK → login con password nueva funciona, con la vieja 401, y un refresh previo al reset → 401 (sesiones caídas) · token de reset reusado → 400 · password débil en reset → 400 y el token **sigue siendo válido** (no se quema en fallo de validación).
**Change-password**: current errónea → 401 · OK → la sesión actual sigue viva (refresh propio funciona), otra sesión sembrada del mismo user → muerta.
**Cookie**: `Path=/api/v1/auth` — un GET a `/api/v1/exercises` con la cookie no la reenvía (estructural: assert de atributos).

### Criterios de aceptación
- [ ] Escenario de robo (reuso → familia revocada) verde y con log de seguridad verificado en el test.
- [ ] Transacción marca+inserta demostrada (test que fuerza fallo del insert → el token original NO queda revocado).
- [ ] Los 5 endpoints en `@bv/contracts` con schemas.
- [ ] Matriz completa de casos verde.

## F1-06 ⛔ — Middlewares de autorización + rate limiting

**Dificultad**: 🔴 Senior · **Modelo**: Opus · **Depende de**: F1-04
**Docs**: [Arquitectura §5](../02-arquitectura.md) · [Seguridad §1-2](../05-seguridad.md) · RN-03/04

### Especificación técnica

**`requireAuth`**: extrae `Authorization: Bearer` → verifica JWT (firma + exp). Token expirado → `TOKEN_EXPIRED` 401 (código **distinto** de `TOKEN_INVALID`: el FE dispara refresh solo ante `TOKEN_EXPIRED`). Set `userId` en contexto.

**`tenantGuard`** (después de `requireAuth`): header `X-Org-Id` requerido → falta: `ORG_HEADER_MISSING` 400; no-ObjectId: `VALIDATION_ERROR`. Lookup `memberships.findOne({orgId, userId})`: inexistente **o** `status != 'active'` → `NOT_A_MEMBER` 403 (mismo código para `disabled`/`invited`: no filtrar estado a un no-miembro). Set `org = { orgId, role, membershipId }`.

**`requireRole(action)`**: recibe una **acción** de la matriz `PERMISSIONS` de `@bv/contracts` (no roles sueltos — la matriz es la fuente): `can(role, action)` falso → `FORBIDDEN_ROLE` 403.

**Rate limiting** (`middleware/rate-limit.ts`): ventana fija en Mongo — colección `rateLimits` `{ _id: '<scope>:<key>:<windowStart>', count, expiresAt }` con TTL; `findOneAndUpdate` upsert `$inc count`; si `count > limit` → `RATE_LIMITED` 429 + header `Retry-After` (segundos hasta fin de ventana). Sobrevive deploys y funciona con N réplicas (DEC del doc de escalabilidad).
Factory: `rateLimit({ scope, keyBy: 'ip' | 'body-email' | 'user', limit, windowSec })`. IP real respetando `TRUST_PROXY` (v1 pattern).

Aplicación inicial:

| Endpoint | Límites |
|---|---|
| login | 10 / 15 min por IP **y** 5 / 15 min por email |
| register | 5 / h por IP |
| forgot-password, resend-verification | 3 / h por email + 10 / h por IP |
| refresh | 60 / h por IP |
| orgs/join | 10 / h por usuario |

### Casos de prueba
**requireAuth**: sin header → 401 `TOKEN_INVALID` · token expirado (fake timers) → 401 `TOKEN_EXPIRED` · firma inválida → `TOKEN_INVALID` · válido → pasa con `userId`.
**tenantGuard**: sin header → 400 · org inexistente → 403 `NOT_A_MEMBER` · membership `disabled` → 403 (RN-03) · `invited` → 403 · `active` → pasa con `role` correcto.
**requireRole**: athlete sobre `members:manage` → 403 · admin → pasa · owner → pasa en TODAS las acciones (iterar matriz) · acción inexistente → error de compilación (tipos), no runtime.
**rate limit**: límite 3 con ventana 2 s → requests 1-3 pasan, 4ª → 429 con `Retry-After` · tras la ventana → pasa de nuevo · claves independientes (IP A llena su cupo, IP B pasa) · doble instancia simulada (dos `app` contra la misma DB) comparten contador.

### Criterios de aceptación
- [ ] Trío de middlewares exportado y documentado con ejemplo de uso en el README del módulo.
- [ ] Límites aplicados a los endpoints de la tabla (integración: el 6º login errado consecutivo devuelve 429, no 401).
- [ ] `Retry-After` presente y correcto en 429.
- [ ] Matriz de casos completa verde.

## F1-07 ⛔ — Módulo `orgs` (+ `/me`)

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F1-06
**Docs**: RN-01/02/24 · [Funcional F1/F2/F7](../01-funcional.md) · [Arquitectura §4](../02-arquitectura.md)

### Especificación técnica

| Método y ruta | Guard | Descripción |
|---|---|---|
| `POST /api/v1/orgs` | auth + email verificado | Crea org + membresía `owner`. Body `{ name: 2..60, timezone }` |
| `GET /api/v1/orgs/current` | tenant | Org DTO (el `joinCode` solo si role admin/owner) |
| `PATCH /api/v1/orgs/current` | tenant + `org:settings` (owner) | `{ name?, timezone?, settings? }` — `cancellationWindowHours: 0..72`, `sessionGenerationDays: 7..60` |
| `POST /api/v1/orgs/current/regenerate-code` | tenant + `org:regenerate-code` (owner) | Nuevo `joinCode`; el anterior muere al instante (RN-01) |
| `POST /api/v1/orgs/join` | auth + verificado | Body `{ code }` (normaliza lowercase/trim) |
| `GET /api/v1/me` | auth | UserDto |
| `GET /api/v1/me/memberships` | auth | Para el selector de org (F7): `[{ orgId, orgName, orgSlug, role, status }]` — solo `active`/`invited` |

**Reglas**:
- `timezone`: validar con `Intl` — `supportedValuesOf('timeZone')` como fast-path **más** probe del constructor `Intl.DateTimeFormat` (supportedValuesOf devuelve canónicos CLDR y excluye alias válidos como `America/Argentina/Buenos_Aires`).
- `slug`: slugify del nombre (`bahía cross` → `bahia-cross`); colisión → sufijo `-2`, `-3`.
- `joinCode`: `<slug-truncado-a-24>-<4 chars [a-z0-9]>` — cumple RN-01 (`[a-z0-9-]{4,32}`, único global, case-insensitive: se guarda lowercase).
- **Join**: buscar org por `joinCode`; inexistente → `ORG_CODE_INVALID` 404. Si existe membresía previa del user en esa org → `ALREADY_MEMBER` 409 (incluida `disabled`: no permite auto-reactivarse). Si existe una membresía `invited` con `invitedEmail == user.email` → **vincular**: set `userId`, `status:'active'`, `joinedAt`, conservar `profile`/`adminNotes` (flujo F2 del Funcional — evita fichas duplicadas). Si no → crear membresía `athlete` `active`. Carrera de doble join → E11000 del índice `{orgId,userId}` → mapear a `ALREADY_MEMBER`.
- Al crear org: settings con defaults (`cancellationWindowHours: 2`, `sessionGenerationDays: 14`), `status: 'active'`.

### Casos de prueba
**Crear org**: feliz → org + membership owner en DB · sin email verificado → 403 · timezone inválida (`America/Springfield`) → 400 · dos orgs con el mismo nombre → slugs `x` y `x-2`.
**Settings**: admin (no owner) intenta PATCH → 403 `FORBIDDEN_ROLE` · `cancellationWindowHours: 100` → 400 · owner cambia timezone → reflejado en GET.
**Regenerate**: código viejo → join 404 · miembros existentes intactos (RN-01) · admin no puede (403).
**Join**: feliz athlete · case-insensitive (`BAHIA-CROSS-x9k2` funciona) · código inexistente → 404 · segundo join → 409 · join con membresía `disabled` previa → 409 (no reactiva) · **vinculación**: member invitado con `invitedEmail` + `adminNotes` sembradas → join → misma membership (id estable), notas conservadas, `status: active` · dos joins concurrentes (`Promise.all`) → exactamente una membership.
**/me/memberships**: user en 2 orgs (1 active, 1 invited, 1 disabled) → devuelve 2 (disabled excluida) con nombres de org resueltos (`$lookup`).

### Criterios de aceptación
- [ ] Tabla de endpoints exacta; schemas en `@bv/contracts` (`createOrgBody`, `orgDto`, `joinOrgBody`, `membershipSummaryDto` reusado).
- [ ] `joinCode` nunca visible para role `athlete` (test del DTO).
- [ ] Matriz de casos completa verde, incluida la vinculación y la carrera de join.

## F1-09 ⛔ — Suite de aislamiento multi-tenant

**Dificultad**: 🔴 Senior · **Modelo**: Fable · **Depende de**: F1-07, F1-08
**Docs**: [Testing §5](../08-testing.md) · [Seguridad §2](../05-seguridad.md)

### Objetivo
Harness permanente que garantiza que **ningún endpoint presente o futuro** filtra datos entre organizaciones ni salta la matriz de roles. Es infraestructura de test, no una suite puntual.

### Especificación técnica

**Registro de políticas** (`apps/api/src/route-policies.ts`):
```ts
export const ROUTE_POLICIES: RoutePolicy[] = [
  { method: 'POST', path: '/api/v1/auth/login',    access: 'public' },
  { method: 'GET',  path: '/api/v1/me',            access: 'user' },
  { method: 'GET',  path: '/api/v1/members/:id',   access: 'admin', resource: 'membership' },
  // ... TODAS las rutas
];
// access: 'public' | 'user' (solo auth) | 'member' | 'admin' | 'owner'
// resource: clave del catálogo de factories para el test de IDOR cross-org
```

**Test 1 — completitud**: introspección de rutas registradas en Hono (`app.routes`) filtrando `/api/v1/*` → toda ruta debe tener exactamente una política. Ruta sin política o política huérfana → test rojo con mensaje "registrá la ruta en route-policies.ts". *Este test convierte el olvido en fallo de build.*

**Test 2 — matriz de acceso** (generado por `describe.each` sobre las políticas):
- Setup una vez: `orgA`, `orgB`, usuarios: `ownerA`, `adminA`, `athleteA` (miembros de A), `athleteB` (solo B), `outsider` (sin orgs), `disabledA` (membership disabled en A).
- Para `access: 'member'|'admin'|'owner'`: sin token → 401 · `outsider` con `X-Org-Id: A` → 403 · `disabledA` → 403 · `athleteA` sobre rutas `admin` → 403 · `adminA` sobre rutas `owner` → 403.

**Test 3 — IDOR cross-org**: para políticas con `resource`: una **factory por recurso** (`factories.ts`: `membership`, y F2+ suman `exercise`, `entry`, `template`, `session`, `pack`, `assignment`, `booking`) crea el recurso en `orgB`; se llama el endpoint como `adminA` con `X-Org-Id: A` y el `:id` de B → **404** esperado (no 403, no 200 — no revelar existencia).

Requests con bodies mínimos válidos por ruta (la política puede declarar `sampleBody`) — el objetivo es pasar la validación Zod y morir en authz, no antes.

### Casos de prueba
La suite ES los casos. Verificación de la suite misma:
1. Agregar una ruta de prueba sin política → Test 1 rojo (luego se quita).
2. Sabotaje controlado: comentar el filtro `orgId` en un repo (en un branch de prueba) → Test 3 rojo. Documentar la corrida en el PR — demuestra que la red atrapa el bug real.

### Criterios de aceptación
- [ ] Toda ruta existente al momento del merge tiene política y pasa su matriz.
- [ ] El sabotaje del caso 2 fue ejecutado y atrapado (evidencia en el PR).
- [ ] README del harness: cómo registrar una ruta nueva y cómo agregar una factory (3 pasos, para que cualquier tarea futura lo haga sin leer el código del harness).
- [ ] Corre dentro de `turbo run test` normal (sin flag especial — si es opcional, se va a olvidar).

## F1-10 — Job scheduler + `expire-packs`

**Dificultad**: 🟡 Semi-senior · **Modelo**: Sonnet · **Depende de**: F1-02
**Docs**: [Arquitectura §8](../02-arquitectura.md) · DEC-09 · RN-13/18

### Especificación técnica

**`jobs/scheduler.ts`**: registro de jobs con `node-cron`. Wrapper por job: log inicio/fin/duración/conteos, `try/catch` (error → log error, nunca tumba el proceso), flag in-process anti-solapamiento (si la corrida anterior sigue viva, la nueva se saltea con log warn). Arranca en `index.ts` solo si `ENABLE_JOBS !== 'false'` (env — permite réplicas sin jobs a futuro).

**`jobs/expire-packs.ts`** (cron `0 * * * *`):
```ts
const res = await packAssignments().updateMany(
  { status: 'active', expiresAt: { $lt: new Date() } },
  { $set: { status: 'expired', updatedAt: new Date() } },
);
// idempotente: el filtro por status hace que re-correr no toque nada
```
Recordatorio arquitectónico (documentar en el código): **la lectura nunca confía en el job** — todo consumidor de packs valida `expiresAt > now` además del status ([Arquitectura §8](../02-arquitectura.md)).

### Casos de prueba
1. Assignment `active` con `expiresAt` ayer → job → `expired`; `updatedAt` actualizado.
2. `active` que vence mañana → intacto.
3. `cancelled` con fecha pasada → intacto (solo transiciona `active`).
4. Doble corrida consecutiva → segunda reporta 0 modificados.
5. Job que lanza excepción (mock) → proceso vivo, error logueado.
6. Solapamiento: corrida lenta simulada + trigger nuevo → segunda se saltea con warn.

### Criterios de aceptación
- [ ] Scheduler con wrapper estándar reutilizable (F3-01 registra el suyo sin tocar el scheduler).
- [ ] 6 casos verdes con fake timers.
- [ ] Logs de job con formato uniforme (`job`, `durationMs`, `modified`).
