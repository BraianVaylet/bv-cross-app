## F3-01 ⛔ — Módulo `schedule`: templates + materialización + sesiones

**Dificultad**: 🔴 Senior · **Modelo**: Fable · **Depende de**: F1-06, F1-10 · **Bloquea**: F3-06, F4-01
**Docs**: RN-05/09/10 · DEC-07/08/09 · [Arquitectura §7 (timezones — leer COMPLETO antes de empezar)](../02-arquitectura.md) · [Arquitectura §8](../02-arquitectura.md)

### Objetivo
La grilla semanal del gimnasio como plantillas recurrentes, materializadas en sesiones concretas reservables. Es la tarea con más lógica de fechas del proyecto.

### Especificación técnica

**Templates** (guard `schedule:manage`):

| Método y ruta | Descripción |
|---|---|
| `POST /templates` | `{ weekday: 0..6, startTime: 'HH:mm', durationMin: 15..240, discipline: 1..40, description? ≤200, capacity: 1..100 }` |
| `GET /templates` | Lista (activos primero, orden weekday+startTime) |
| `PATCH /templates/:id` | Mismos campos. Efecto sobre sesiones: ver "propagación" |
| `DELETE /templates/:id` | Ver "borrado" |

- **Solapamiento**: dos templates activos del mismo weekday cuyos rangos `[startTime, startTime+duration)` se pisan → permitido **con warning** en la respuesta (`details.overlaps`) — hay gimnasios con clases paralelas; el CRM lo muestra, no lo bloquea.
- **Propagación de PATCH (RN-05)**: se regeneran (borrar + rematerializar) solo las sesiones futuras del template **con `bookedCount = 0`**; las que tienen reservas quedan como están (el admin las ajusta una a una si quiere).
- **Borrado**: el template se elimina; sesiones futuras con `bookedCount = 0` se borran; con reservas → quedan (conservan `templateId` huérfano; se administran manualmente). Respuesta informa `{ deletedSessions, keptSessions }`.

**Job `materialize-sessions`** (cron `0 * * * *`, registrado en el scheduler F1-10):
```
por cada org activa:
  horizon = hoy(org.tz) + settings.sessionGenerationDays
  por cada template activo:
    por cada fecha en [hoy, horizon] con weekday coincidente (calculado en org.tz):
      startsAtUtc = tzToUtc(fecha + startTime, org.timezone)   // @date-fns/tz
      si startsAtUtc > now: insertOne({ ...copia de template, startsAt, endsAt, bookedCount: 0,
                                        status: 'scheduled' })  // E11000 por índice {templateId, startsAt} → ignorar
```
- Idempotencia por índice único — el job puede correr N veces o en N réplicas sin duplicar (DEC-09).
- La sesión **copia** `capacity/discipline/description` (DEC-07: editable por sesión sin tocar el template).
- DST: la conversión usa la timezone en **esa fecha** (no un offset fijo). Horario inexistente por salto DST → se usa el resultado determinista de la librería (documentado en el código con el caso).

**Sesiones**:

| Método y ruta | Guard | Descripción |
|---|---|---|
| `GET /sessions?from&to` | member | Rango ≤ 31 días (más → `VALIDATION_ERROR`). Incluye `bookedCount`, `capacity`, `myBookingId` (si el caller tiene reserva `booked`) |
| `POST /sessions` | admin | Sesión manual (sin template): mismos campos + `date` |
| `PATCH /sessions/:id` | admin | `capacity` (≥ `bookedCount`, si no `CAPACITY_BELOW_BOOKED` 409), `description` |
| `POST /sessions/:id/cancel` | admin | RN-09. **Hasta F4-01**: solo si `bookedCount = 0`, si no `HAS_BOOKINGS` 409 (F4-01 reemplaza este guard por la devolución transaccional de créditos) |
| `GET /sessions/:id/attendees` | admin | Reservas `booked` con nombre del miembro y hora de reserva |

### Casos de prueba
**Timezone (los más importantes)**:
1. Template lunes 18:00, org AR (UTC-3, sin DST) → sesión `startsAt = 21:00Z` del lunes correcto.
2. Org `Europe/Madrid`, template 02:30, materializando la fecha del salto de primavera → resultado determinista de la librería, sin crash, sin duplicado (test con fecha real de cambio DST).
3. Template domingo 00:30 org AR → cae el domingo local (lunes 03:30Z) — el weekday se evalúa en tz org, no UTC.
**Job**: 4. doble corrida → 0 duplicados · 5. horizonte respeta `sessionGenerationDays` por org (2 orgs con valores distintos) · 6. sesión ya iniciada nunca se crea retroactivamente · 7. template desactivado deja de materializar; sus sesiones futuras existentes quedan.
**Propagación**: 8. PATCH capacity con 3 sesiones futuras (1 con reserva simulada `bookedCount>0`) → 2 regeneradas, 1 intacta · 9. DELETE ídem → `{deletedSessions: 2, keptSessions: 1}`.
**Sesiones**: 10. rango de 45 días → 400 · 11. `PATCH capacity` por debajo de `bookedCount` → 409 · 12. cancel con reservas → 409 `HAS_BOOKINGS` (pre-F4) · 13. `myBookingId` presente solo para el que reservó (fixture con booking manual).
**Aislamiento**: factories `template` y `session` registradas → suite F1-09.

### Criterios de aceptación
- [ ] Los 13 casos verdes; los 3 de timezone revisados con especial atención en el PR (pegar los valores UTC esperados a mano).
- [ ] Job integrado al scheduler con wrapper estándar (logs de conteos por org).
- [ ] Seed extendido: grilla lun–sáb materializada 14 días (F1-11).
- [ ] Contracts: `templateDto`, `sessionDto`, bodies. Política + factories en F1-09.

## F3-02 ⛔ — Módulo `packs` (catálogo)

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F1-06 · **Bloquea**: F3-03
**Docs**: RN-14/15/16 · [Funcional §5 packs](../01-funcional.md)

### Especificación técnica

Guard `packs:manage` en mutaciones; `GET` para member (las apps atleta muestran el catálogo? **No** — fase 1 el atleta solo ve SUS asignaciones; `GET /packs` queda admin-only, se relaja si el roadmap de MP lo pide).

| Método y ruta | Descripción |
|---|---|
| `GET /packs?includeArchived=` | Lista catálogo |
| `POST /packs` | `{ name: 1..60, classCount: 1..100, durationDays: 1..365, price: int ≥ 0 (pesos ARS, sin centavos), paymentMethod, internalNotes? ≤500 }` — `currency` fija `'ARS'` server-side |
| `PATCH /packs/:id` | Ver matriz de campos |
| `DELETE /packs/:id` | Solo si **nunca** tuvo asignaciones; si tuvo (cualquier estado) → `PACK_IN_USE` 409 con mensaje "archivalo en su lugar" |
| `POST /packs/:id/archive` · `/restore` | RN-15: archivado no asignable, visible en historial |

**Matriz de edición (RN-14)** — el corazón de la tarea:

| Campo | Sin asignaciones `active` | Con asignaciones `active` |
|---|---|---|
| `name`, `internalNotes` | ✅ | ✅ (no afecta al cliente: su snapshot ya congeló el nombre — RN-16) |
| `price`, `classCount`, `durationDays`, `paymentMethod` | ✅ | ❌ `PACK_IN_USE` 409 con `details.activeAssignments: n` |

`hasActiveAssignments(packId)`: `countDocuments({ packId, status: 'active' })` — se evalúa **en la misma operación** que el update (patrón: validar y actualizar; la ventana de carrera entre count y update es aceptable acá porque el snapshot RN-16 protege al cliente incluso si se cuela — anotarlo en el código).

### Casos de prueba
1. CRUD feliz; `currency` siempre `'ARS'` aunque el body intente otra (`.strict()` la rechaza).
2. `price: 25000.5` → 400 (entero) · `durationDays: 0` → 400.
3. Matriz: pack con asignación `active` → PATCH `price` 409 con conteo en details; PATCH `name` OK · misma prueba con asignación `expired` → todo editable (solo `active` bloquea).
4. DELETE de pack con asignación histórica (`expired`) → 409 sugiere archivar · DELETE de pack virgen → 200 y desaparece.
5. Archive → no asignable (test cruzado en F3-03) pero visible con `includeArchived=1` · restore lo reactiva.
6. Aislamiento: factory `pack` → suite F1-09.

### Criterios de aceptación
- [ ] Matriz RN-14 implementada exactamente y testeada campo por campo.
- [ ] Seed extendido: 2 packs (F1-11).
- [ ] Contracts + políticas al día.

## F3-03 ⛔ — Módulo `assignments` (packs asignados + pago manual)

**Dificultad**: 🔴 Senior · **Modelo**: Fable · **Depende de**: F3-02 · **Bloquea**: F4-01
**Docs**: RN-12/13/16/17/18 · [Funcional F3](../01-funcional.md) · [Arquitectura §4 `packAssignments`, §7](../02-arquitectura.md)

### Objetivo
El "saldo" del atleta: asignación con snapshot inmutable, vencimiento correcto en la timezone del gimnasio y registro del pago manual.

### Especificación técnica

| Método y ruta | Guard | Descripción |
|---|---|---|
| `POST /members/:id/assignments` | `assignments:manage` | Crear asignación (ver reglas) |
| `GET /members/:id/assignments?status=` | admin | Historial del miembro, orden `createdAt desc` |
| `GET /me/assignments?status=` | member | Las del caller en la org activa (base de F4-06) |
| `POST /assignments/:id/cancel` | admin | Body `{ reason: 1..300 }` → `status:'cancelled'` |

**Reglas de creación**:
- Body: `{ packId, startsAt?: calendarDate (default: hoy en tz org), payment: { amount?: int (default snapshot.price — editable: descuentos), method? (default del pack), paidAt?: isoDate (default now), notes? ≤300 } }`.
- Validaciones: pack existe y no archivado (`PACK_ARCHIVED` 409 — nuevo código a agregar en contracts) · miembro existe, `status:'active'` y no owner-check necesario (`MEMBER_DISABLED` 409 si disabled; `invited` **sí** puede recibir pack: el gimnasio carga todo antes de que el atleta se registre) · `startsAt` dentro de ±1 año (typos).
- **Snapshot (RN-16)**: copia literal `{ name, classCount, durationDays, price, currency, paymentMethod }` del pack al momento de crear. Inmutable para siempre.
- **Vencimiento (RN-18)**: `expiresAt = fin del día (23:59:59.999, tz org) de (startsAt + durationDays)`. Nota de diseño: la vigencia incluye el día de inicio completo y termina `durationDays` días después — decisión RN-18, generosa a favor del cliente; documentada en el código.
- `classesUsed: 0`, `status: 'active'` (aunque `startsAt` sea futuro: un pack "arranca el lunes" es visible y válido; **no** usable para reservar hasta `startsAt` — regla que consume F4-01: filtro `startsAt <= now`). RN-17: múltiples activos permitidos, sin validación de solapamiento.
- **Cancel**: solo desde `active` → `cancelled` + `cancelledReason`. Reservas futuras hechas contra ese pack **quedan** (se resuelven si el atleta las cancela — el crédito vuelve a un pack cancelado = inutilizable; documentado). `expired`/`exhausted` no se cancelan (409, son terminales — RN-13).

### Casos de prueba
1. Feliz con defaults: `payment.amount == snapshot.price`, `paidAt ≈ now`, `expiresAt` correcto.
2. **Snapshot**: crear asignación → PATCH del pack (name+price cuando sea editable / archivarlo) → la asignación no cambia ni un byte (assert profundo).
3. **Timezone del vencimiento**: org AR, `startsAt: '2026-07-01'`, `durationDays: 30` → `expiresAt = 2026-07-31T23:59:59.999-03:00` = `2026-08-01T02:59:59.999Z` (valor UTC exacto en el assert).
4. Cruce de mes/año: `startsAt: '2026-12-15'` + 30 → 14/01/2027 fin de día AR.
5. `payment.amount: 20000` sobre pack de 25000 → registrado 20000 (descuento).
6. Pack archivado → 409 `PACK_ARCHIVED` · miembro `disabled` → 409 `MEMBER_DISABLED` · miembro `invited` → **OK**.
7. RN-17: segunda asignación activa solapada al mismo miembro → OK; `GET /me/assignments` las trae ordenadas.
8. Cancel: feliz con razón → `cancelled` · cancel de `expired` → 409 · doble cancel → 409.
9. `GET /me/assignments` de atleta B no ve las de A (y suite F1-09 con factory `assignment`).
10. Job F1-10 integra: asignación activa con `expiresAt` pasado → `expired` (test cruzado ya existente — solo verificar que el seed de este módulo lo alimenta).

### Criterios de aceptación
- [ ] Los 10 casos verdes; el #3 con el instante UTC calculado a mano en el assert (no derivado con la misma librería — evita test tautológico).
- [ ] Contracts: `assignmentDto` (incluye `remaining = snapshot.classCount - classesUsed` computado), bodies. Políticas + factory.
- [ ] Seed extendido: 4 asignaciones en estados distintos (F1-11).

## F3-10 — Módulo `stats` fase 1 + Dashboard del CRM

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F3-05..09 (datos), F1-06
**Docs**: [Funcional §6 dashboard](../01-funcional.md) · [Escalabilidad §3-4](../07-escalabilidad.md) (sin colecciones nuevas)

### Especificación técnica

**API — `GET /stats/dashboard`** [admin], una sola llamada que agrega:

| Bloque | Contenido | Fuente |
|---|---|---|
| `today.sessions` | sesiones de hoy (tz org): hora, disciplina, `booked/capacity` | `classSessions` rango del día |
| `week` | reservas y cancelaciones de la semana en curso | `bookings` (vacío pre-F4, contrato ya definido) |
| `expiringAssignments` | asignaciones activas que vencen en ≤7 días: miembro, vence, saldo | `packAssignments` índice `{orgId,status,expiresAt}` |
| `inactiveMembers` | miembros activos sin reservas hace ≥14 días | `memberships` − `bookings` (pre-F4: lista vacía) |
| `revenue.month` | suma de `payment.amount` de asignaciones creadas en el mes calendario (tz org) | `packAssignments` |
| `newMembers.month` | membresías activadas en el mes | `memberships.joinedAt` |

- Cada bloque = un pipeline separado corrido con `Promise.all`; presupuesto **p95 total < 300 ms con seed ×100**.
- Script `apps/api/scripts/loadgen.ts`: multiplica el seed ×N (orgs sintéticas con volumen realista de [Escalabilidad §0](../07-escalabilidad.md)) — herramienta permanente para presupuestos de perf.
- Sin colección de snapshots (regla anti-sobre-ingeniería; el disparador para agregarla está en Escalabilidad §2).

**FE — `/` (dashboard)**: `StatCard` (nuevo en `@bv/ui`: valor grande en serif display, label, delta opcional) para ingresos del mes / nuevos miembros / reservas de semana · lista "Hoy" con ocupación (click → detalle de sesión en Clases) · lista "Vencen esta semana" (click → ficha del cliente) · lista "Sin actividad 14+ días" (click → ficha). Cada lista con empty state positivo ("Nadie por vencer 🎉").

### Casos de prueba
1. Fixtures deterministas (fechas fijas con fake clock): dashboard devuelve números **calculados a mano** en el test (ej.: 3 asignaciones de $25.000/$32.000/$20.000 en el mes + 1 fuera → `revenue.month = 77000`).
2. Frontera de mes en tz org: asignación creada el 01/07 00:30 AR (30/06 UTC) cuenta en julio (test explícito del pipeline con tz).
3. `expiringAssignments`: vence en 6 días → aparece; 8 días → no; vencida → no.
4. Perf: `loadgen ×100` + medición del endpoint < umbral (mismo patrón que F3-09).
5. Component: cards y listas con datos mock; navegación por click correcta.

### Criterios de aceptación
- [ ] Números validados a mano (no snapshot del propio código).
- [ ] `loadgen.ts` commiteado y documentado.
- [ ] p95 medido y anotado en el PR.
- [ ] `StatCard` en `@bv/ui`.
