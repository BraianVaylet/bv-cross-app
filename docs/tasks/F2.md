## F2-01 ⛔ — Módulo `exercises`

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F1-06, F1-09
**Docs**: RN-19/20/21/23 · [Arquitectura §4](../02-arquitectura.md) (colección `exercises`)

### Objetivo
Una colección, dos alcances: catálogo de la organización (gestionado por admins, visible para todos sus miembros) y ejercicios personales (privados del atleta, cross-org).

### Especificación técnica

| Método y ruta | Guard | Descripción |
|---|---|---|
| `GET /exercises?scope=org\|personal\|all&includeArchived=` | member | `all` default. Devuelve catálogo de la org activa (sin archivados salvo `includeArchived=1`, solo admin) + personales del usuario (cross-org, RN-21) |
| `POST /exercises` | member | Admin puede `scope:'org'`; athlete **siempre** se fuerza a `scope:'personal'` (si manda `org` → 403 `FORBIDDEN_ROLE`) |
| `GET /exercises/:id` | member | Org: cualquier miembro. Personal: solo el dueño (ajeno → 404) |
| `PATCH /exercises/:id` | member | Org: solo `exercises:manage-catalog`. Personal: solo dueño |
| `POST /exercises/:id/archive` · `/restore` | admin | Solo `scope:'org'` (RN-19) |
| `DELETE /exercises/:id` | member | Solo `scope:'personal'` y solo el dueño; **cascada**: borra sus `rmEntries` (comportamiento v1) |

**Campos** (`@bv/contracts`): `name 1..80` · `discipline? 1..40` · `type: 'weight'|'reps'` · `imageUrl? url https` · `notes? ≤500`.
**Reglas**:
- `type` **inmutable** si el ejercicio tiene entries (`TYPE_LOCKED` 409) — cambiarlo invalidaría el historial (kg vs reps).
- Documento: `scope:'org'` ⇒ `orgId` set / `ownerUserId: null`; `scope:'personal'` ⇒ inverso. Invariante validada en el repo (assert) además del schema.
- Unicidad blanda de nombre en catálogo: crear con nombre ya existente (case-insensitive) en la misma org → `VALIDATION_ERROR` con mensaje claro (evita catálogos con "Back Squat" ×3). Personales: sin restricción.
- Los personales NO llevan `X-Org-Id` en su filtro (RN-21) pero el endpoint igual exige `tenantGuard` (contexto de app siempre tiene org activa).

### Casos de prueba
1. Admin crea org-exercise → visible en `GET` de otro miembro; athlete crea sin `scope` → nace `personal`; athlete pidiendo `scope:'org'` → 403 `FORBIDDEN_ROLE` (regla de la tabla — el 403 explícito le avisa al FE en vez de coercionar en silencio).
2. Personal de atleta A: invisible para atleta B **y para el admin** (RN-20) → `GET /exercises/:id` ajeno = 404.
3. Cross-org: catálogo de org B invisible con `X-Org-Id: A` (suite F1-09 + factory `exercise`).
4. Archivar: desaparece del `GET` default, aparece con `includeArchived=1` (admin); las entries de atletas sobre él siguen legibles (RN-19).
5. `TYPE_LOCKED`: crear weight → agregar entry → PATCH `type: reps` → 409; sin entries → permitido.
6. DELETE personal con 3 entries → ejercicio y entries desaparecen; DELETE de org-exercise → 403/405.
7. Nombre duplicado en catálogo (`back squat` vs `Back Squat`) → 400; el mismo nombre como personal → OK.
8. Athlete ve catálogo + solo SUS personales en `all` (test con 2 atletas mezclados).

### Criterios de aceptación
- [ ] Tabla de endpoints exacta; DTOs en `@bv/contracts` (`exerciseDto`, `createExerciseBody`, `updateExerciseBody`).
- [ ] Política + factory registradas en F1-09.
- [ ] Los 8 casos verdes.
- [ ] Seed extendido: 12 ejercicios de catálogo (10 weight + 2 reps) — ver F1-11.

## F2-02 ⛔ — Módulo `entries` (registros de carga)

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F2-01
**Docs**: RN-20/21/22/23 · [Arquitectura §4](../02-arquitectura.md) (`rmEntries`) · comportamiento v1 (última entry no se borra)

### Especificación técnica

| Método y ruta | Guard | Descripción |
|---|---|---|
| `POST /entries` | member | `{ exerciseId, kg? \| reps?, date: calendarDate, comment? ≤300, painFlag? }` |
| `GET /entries?exerciseId=&after=&limit=` | member | Solo entries propias; orden `date desc, _id desc` (criterio RN-22) |
| `DELETE /entries/:id` | member | Solo el dueño; última entry del ejercicio → `LAST_ENTRY` 409 (regla v1) |
| `GET /members/:id/entries?exerciseId=` | admin | Entries del miembro **solo sobre ejercicios de catálogo** (RN-20) |

**Reglas**:
- Acceso al ejercicio: personal → dueño; org → miembro de esa org. Ejercicio archivado → se puede **leer** historial pero no crear entry nueva (`VALIDATION_ERROR` con mensaje "ejercicio archivado").
- **XOR de medida** (RN-23): `kg` requerido y `reps` prohibido si `exercise.type === 'weight'`; inverso para `reps`. Violación → `VALIDATION_ERROR` con code interno `WRONG_MEASURE` en `details`.
- Rangos sanidad: `kg` 0.5..600 (decimal, hasta 2 decimales) · `reps` entero 1..500.
- `date` no futura **en la timezone de la org** (una carga "de hoy" a las 23:30 AR no debe rechazarse porque en UTC ya es mañana).
- Stamping: `entry.orgId = exercise.scope === 'org' ? exercise.orgId : null` (RN-21) — habilita las stats del CRM por org sin tocar personales.
- RM vigente (RN-22) = primera según el orden definido; **no** se persiste flag: se deriva (el FE y las stats usan el mismo criterio de orden).

### Casos de prueba
1. Feliz weight y feliz reps; `kg` en ejercicio reps → 400 `WRONG_MEASURE` (y viceversa).
2. `kg: 0` → 400 · `reps: 2.5` → 400 · `date: '2026-13-01'` → 400 · fecha futura → 400 · fecha "hoy 23:30 AR / mañana UTC" → **aceptada** (test con clock fijo).
3. RN-22: entries con fechas [10/06, 25/06, 20/06] → vigente 25/06; dos con la misma fecha → gana la de `_id` mayor (creada después).
4. `LAST_ENTRY`: ejercicio con 1 entry → DELETE 409; con 2 → borra una OK.
5. Atleta A no lee ni borra entries de B (404).
6. Admin: `GET /members/:id/entries` devuelve las de catálogo y **omite** personales (fixture mixta, RN-20); admin de otra org → 404 (suite).
7. Entry sobre ejercicio archivado → 400 con mensaje de archivado; historial sigue legible.
8. `painFlag: true` persistido y presente en el DTO.
9. Entry sobre ejercicio personal → `orgId: null` en DB; sobre catálogo → `orgId` de la org (assert directo).

### Criterios de aceptación
- [ ] Tabla + reglas implementadas; DTOs en contracts; política + factory en F1-09.
- [ ] 9 casos verdes.
- [ ] Seed extendido: historial 6 meses para atletas 1-3 (ver F1-11).

## F2-03 ⛔ — bv-cross FE: migración al monorepo + auth nueva

**Dificultad**: 🔴 Senior · **Modelo**: Opus · **Depende de**: F0-03, F1-05, F1-07 · **Bloquea**: F2-04..07
**Docs**: [Arquitectura §5](../02-arquitectura.md) · [Técnico §3](../03-tecnico.md) (scaffolding FE) · [Funcional F7](../01-funcional.md)

### Objetivo
La app v1 viviendo en `apps/cross`, hablando con la API nueva: auth por email, refresh silencioso, organización activa. Las pantallas de negocio (Home/Detalle/Calculadora) se tocan lo mínimo.

### Alcance
**Incluye**: copia del código v1, capa `auth/` y `api/` nuevas, pantallas de auth nuevas, selector de org, adopción de `@bv/ui` y `@bv/contracts`.
**No incluye**: pantalla de join (F2-04), Home con catálogo (F2-05), cuenta/PWA (F2-06).

### Especificación técnica

**Migración física**: copiar `bv-cross/client/*` → `apps/cross/`; `package.json` renombrado `@bv/cross`, deps `workspace:*` (`@bv/ui`, `@bv/contracts`, `@bv/config`); borrar `ui.tsx` y tokens locales → imports de `@bv/ui`; `index.css` = `@import '@bv/ui/tokens.css'` + estilos propios mínimos.

**`src/auth/`** — decisiones de diseño obligatorias:
- Access token **en memoria de módulo** (variable en closure de `api/client.ts`), jamás en localStorage/sessionStorage. `AuthContext` expone `{ status: 'loading'|'anon'|'authed', user, memberships }`.
- Al montar la app: `POST /auth/refresh` (la cookie viaja sola) → 200: hidrata sesión sin login; 401: `anon`.
- `OrgContext`: `activeOrgId` con persistencia en `localStorage['bv.activeOrgId']` (el id de org NO es secreto), validado contra `memberships` al hidratar (si ya no es miembro → limpiar y re-seleccionar).

**`src/api/client.ts`** (única puerta a la red):
- `request(path, { method, body, schema })`: base `VITE_API_URL`, `Authorization: Bearer`, `X-Org-Id` (si hay org activa), `credentials: 'include'` **solo** en llamadas `/auth/*`.
- **401 `TOKEN_EXPIRED` → refresh single-flight**: una sola promesa de refresh compartida entre requests concurrentes; reintenta la original **una vez**; si el refresh da 401 → limpiar sesión → `/login`. Otros códigos → lanzar `ApiError { code, message, details }` tipado.
- Respuestas validadas con el schema de `@bv/contracts` en dev (en prod: parse laxo para no romper por campos nuevos — usar `.passthrough()` en dev-check o try/catch con log).

**Rutas**: públicas `/login /register /verify /forgot /reset` · protegidas (requieren `authed`) `/join /select-org` y las de negocio (`/`, `/exercise/:id`, …). Guard: `authed` sin membresías → `/join`; con >1 → `/select-org` si no hay org activa (F7); con 1 → auto-selección.

**Pantallas nuevas** (con `@bv/ui`, textos DS §6): Login · Register (con estado "revisá tu email") · Verify (lee `?token`, POST, feedback + CTA a login) · Forgot/Reset · SelectOrg (cards con nombre de org + rol). **Se eliminan**: pregunta de seguridad, alias, `Recover` v1.

### Casos de prueba
**Unit (Vitest)**:
1. Single-flight: 2 requests concurrentes reciben 401 `TOKEN_EXPIRED` → exactamente **1** llamada a refresh (spy), ambas reintentadas OK.
2. Refresh falla → sesión limpia + redirect state `anon`; el access token en memoria queda `null`.
3. `X-Org-Id` presente cuando hay org activa; ausente en `/auth/*`.
4. `ApiError` mapea el envelope (code/message) correctamente.
5. Selección de org: 0 memberships → `/join`; 1 → auto; 2 → `/select-org`; org guardada inválida → re-selección.
**Manual (checklist en el PR)**: flujo completo contra API local seedeada; recarga de página mantiene sesión (refresh silencioso); logout limpia y las rutas protegidas expulsan.

### Criterios de aceptación
- [ ] `pnpm --filter @bv/cross dev` funcional contra API local; typecheck verde con contracts.
- [ ] Grep sin resultados: `localStorage.*token`, `alias`, `security` (restos de v1).
- [ ] Los 5 unit tests + checklist manual documentado con capturas.
- [ ] Ningún componente genérico local duplicando `@bv/ui` (revisión en PR).

**Docs**: RN-19/20/22 · [Design System §4-5](../04-design-system.md) · UX v1 (Home/Detalle/Calculadora se preservan)

### Especificación técnica
- **Home en dos secciones**: "Ejercicios de {nombre de la org}" (catálogo — el atleta NO puede editarlos ni borrarlos; sin menú de acciones) y "Mis ejercicios" (personales — CRUD completo, UX v1). Búsqueda local única filtrando ambas.
- Imágenes: `imageUrl` del catálogo si existe → fallback al mapa local por nombre (`exerciseImages.ts` v1) → placeholder genérico.
- Detalle/calculadora/historial: sin cambios visuales; el data source pasa a `GET /entries?exerciseId=` con DTOs de contracts. El RM vigente se deriva con el criterio RN-22 (**mismo orden que el server**: helper `currentRm(entries)` compartido en `src/lib` con test).
- Ejercicio de catálogo archivado con historial propio: aparece en una subsección colapsada "Archivados por el gimnasio" (solo lectura — RN-19).
- Estados: skeletons por sección, empty state para "Mis ejercicios" con CTA crear, empty global si la org no cargó catálogo.

### Casos de prueba
1. Unit `currentRm`: fechas desordenadas → gana la más reciente; empate de fecha → mayor `id`; lista vacía → null.
2. Unit agrupador: mezcla org/personal/archivado → 3 buckets correctos.
3. Component: catálogo sin menú de acciones; personal con menú (editar/borrar).
4. Component: búsqueda filtra ambas secciones simultáneamente.
5. Manual: paridad v1 verificada con checklist (crear personal, registrar RM kg y reps, calculadora %s, redondeo configurable, dark mode).

### Criterios de aceptación
- [ ] Checklist de paridad v1 completo y adjunto al PR.

## F2-08 — Migración de datos v1 → v2

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F2-01, F2-02
**Docs**: [Análisis §4](../00-analisis-estado-actual.md) · esquema v1 en [Análisis §1](../00-analisis-estado-actual.md)

### Objetivo
Script one-shot, auditable y reversible, que trae el usuario real de v1 (SQLite) con todo su historial.

### Especificación técnica
`apps/api/scripts/migrate-v1.ts` — CLI: `--sqlite <path> --email <email> --name <nombre> [--commit]`.
- **Dry-run por defecto**: imprime plan (conteos por tabla, mapeos) sin escribir. `--commit` ejecuta.
- Lee v1 con `better-sqlite3` (dep solo de scripts): `users` (toma el único usuario; si hay más, exige `--alias` para elegir), `exercises`, `rm_entries`.
- Mapeo:
  | v1 | v2 |
  |---|---|
  | `users` → | `users` con `email` del CLI, `emailVerifiedAt: now` (el dueño es confiable), password **nueva random** → se fuerza reset por email (no se migra el hash: cambia el esquema de identidad alias→email) |
  | `exercises` → | `exercises` `scope:'personal'`, `ownerUserId`, `type: gimnastico ? 'reps' : 'weight'`, `notes: observacion`, sin org |
  | `exercises.dolor` → | no migra a nivel ejercicio: v2 lo modela por entry (`painFlag`) — se anota en `notes` si estaba activo ("v1: marcado con dolor") |
  | `rm_entries` → | `rmEntries` `{ kg: rm_kg ?? undefined, reps: reps ?? undefined, date, comment, orgId: null }`, `createdAt` preservado |
- Validaciones pre-commit: email no existente en v2 (o `--link-user <id>` para adjuntar a cuenta ya creada) · toda entry referencia un ejercicio migrado · XOR kg/reps consistente con `type` (si v1 tiene sucio: reporta y excluye con warning, no aborta).
- **Verificación post-commit** (el script la imprime): conteos v1 vs v2 por tabla + spot-check: los 3 RMs vigentes de los ejercicios con más historial coinciden.
- **Rollback documentado**: borrar el user creado + sus exercises/entries (comando incluido en el README del script).

### Casos de prueba (Vitest, fixture SQLite generada en el test)
1. Fixture con 2 ejercicios (1 gimnástico) y 5 entries → dry-run reporta 1/2/5, cero escrituras (assert DB vacía).
2. `--commit` → conteos exactos; `gimnastico=1` → `type:'reps'` y entries con `reps`; `observacion` → `notes`.
3. Entry corrupta (rm_kg y reps ambos null) → excluida con warning, resto migra.
4. Re-corrida con `--commit` → aborta ("email ya existe") — no duplica.
5. `--link-user` adjunta a cuenta existente sin tocar sus otros datos.

### Criterios de aceptación
- [ ] Dry-run contra el SQLite real de v1 adjunto al PR (output completo).
- [ ] 5 casos verdes.
- [ ] README del script: uso, mapeos, rollback.
- [ ] La corrida real con `--commit` se hace en F6-02, no acá.
