## F6-01 — Dominios definitivos + SSO verificado

**Dificultad**: 🟢 Junior · **Modelo**: Sonnet · **Depende de**: F0-06, F5 completa
**Docs**: [Deployment §1, §8](../06-deployment.md) · [Arquitectura §5](../02-arquitectura.md)

### Especificación técnica
1. DNS (en el registrador/Cloudflare): 4 registros CNAME → targets de cada servicio.

   | Subdominio | Servicio |
   |---|---|
   | `api.<apex>` | Railway API |
   | `app.<apex>` | FE bv-cross |
   | `agenda.<apex>` | FE bv-cross-schedule |
   | `crm.<apex>` | FE CRM |
2. TLS emitido y forzado en los 4 (el hosting lo gestiona; verificar redirect http→https).
3. Actualizar env y **redeployar**: API (`COOKIE_DOMAIN=.<apex>`, `APP_ORIGINS` con los 3 orígenes finales) + FEs (`VITE_API_URL=https://api.<apex>` — rebuild, es build-time).
4. Emails: dominio verificado en Resend (SPF + DKIM records) para enviar desde `no-reply@<apex>` — sale del sandbox.
5. Ejecutar el [checklist de go-live completo de Deployment §8](../06-deployment.md).

### Casos de prueba (manuales, con evidencia)
1. **SSO — el test central**: login en `app.<apex>` → abrir `agenda.<apex>` → sesión activa sin credenciales → abrir `crm.<apex>` (usuario admin) → activa. DevTools: cookie con `Domain=.<apex>`, `HttpOnly`, `Secure`, `Path=/api/v1/auth`.
2. Logout en una app → tras expirar el access (≤15 min) las otras quedan deslogueadas (el refresh familiar está revocado) — verificar y **documentar esta semántica** (no es logout instantáneo cross-app; decisión aceptada).
3. Email de verificación llega desde `no-reply@<apex>` sin caer en spam (probar Gmail + Outlook).
4. securityheaders.com sobre los 4 dominios (A o mejor).
5. Certificados válidos (SSL Labs A).

### Criterios de aceptación
- [ ] Checklist Deployment §8 completo, ítem por ítem, con evidencia.
- [ ] SSO demostrado con capturas/video.
- [ ] Resend en dominio propio (fuera de sandbox).

## F6-02 — Migración final de datos v1 + apagado de v1

**Dificultad**: 🟢 Junior · **Modelo**: Sonnet · **Depende de**: F2-08, F6-01
**Docs**: [Análisis §4](../00-analisis-estado-actual.md) · script de F2-08

### Especificación técnica — secuencia (es una operación, seguirla en orden)
1. Congelar v1: aviso al usuario (es el dueño del proyecto — coordinación trivial) y no registrar más cargas.
2. Descargar el SQLite del volumen Railway v1 (backup local adicional del archivo).
3. Dry-run del script F2-08 contra producción v2 → revisar el plan impreso.
4. `--commit` → verificación automática de conteos + spot-check manual de 3 RMs vigentes en la app real.
5. El usuario migrado recibe el mail de reset (el script fuerza reset — F2-08), entra, valida su historial.
6. Redirect del dominio/URL v1 → `app.<apex>` (301; servicio estático mínimo o redirect de Railway).
7. Apagar el servicio v1. **Conservar el volumen SQLite 90 días** (recordatorio con fecha en issue). Archivar el repo `bv-cross` v1 en GitHub (read-only, con nota en el README apuntando al monorepo).

### Casos de prueba
1. Verificación de conteos del script (v1 == v2 por tabla) — output adjunto.
2. Login del usuario migrado + spot-check de RMs vigentes (capturas v1 vs v2).
3. URL v1 → redirige a la app nueva.

### Criterios de aceptación
- [ ] Usuario histórico operando en v2 con historial intacto (confirmación explícita).
- [ ] v1 apagada; issue con fecha de borrado del volumen (+90 días).
- [ ] Repo v1 archivado.

## F6-03 — Beta con el primer gimnasio real

**Dificultad**: 🔴 Senior · **Modelo**: humano + Opus (análisis de feedback) · **Depende de**: F6-01
**Docs**: [Funcional §8 (métricas)](../01-funcional.md)

### Objetivo
Validar el producto con un gimnasio real durante 2 semanas, con protocolo de acompañamiento y captura sistemática de fricciones. El éxito de esta tarea define el roadmap (F6-05).

### Especificación técnica — protocolo
1. **Selección**: gimnasio con 30–150 alumnos, dueño accesible y dispuesto a feedback semanal, operación actual por WhatsApp/planilla (dolor real).
2. **Onboarding presencial/videollamada (90 min)**: crear la org con el dueño manejando (no hacerlo por él — es un test de usabilidad) → grilla real → packs reales → carga de 10-20 clientes → envío de invitaciones a un grupo piloto de atletas.
3. **Acompañamiento**: canal WhatsApp directo; revisar Sentry + logs **diariamente** la primera semana; check-in con el dueño (15 min) al final de cada semana con guion (qué usaste, qué te trabó, qué esperabas que hiciera y no hizo).
4. **Registro**: `docs/beta/feedback.md` — tabla: fecha · quién (dueño/atleta) · fricción u observación · severidad (bloqueante/molesta/menor) · acción (issue linkeada / descartada con motivo). TODO entra, aunque parezca menor.
5. **Métricas de la beta** (mirar en datos, no en opiniones): % de reservas por app vs manuales · atletas que registraron ≥1 carga · tiempo del dueño en el CRM · errores en Sentry por sesión.

### Criterios de aceptación (gate del producto, no del código)
- [ ] El gimnasio operó **una semana completa sin intervención nuestra** (cero fixes a mano en DB, cero "lo hago yo por vos").
- [ ] ≥ 70% de las reservas de la semana 2 hechas por los atletas en la app.
- [ ] Feedback log con todas las entradas triageadas (ninguna sin acción/decisión).
- [ ] Bloqueantes encontrados: resueltos y re-verificados con el gimnasio dentro de la beta.

## F6-04 — Runbook de operación

**Dificultad**: 🟡 Semi-senior · **Modelo**: Sonnet · **Depende de**: F6-01 (refleja la infra real)
**Docs**: [Seguridad §9](../05-seguridad.md) · [Deployment](../06-deployment.md) · runbook de restore (F5-07)

### Especificación técnica
`docs/09-runbook.md` — cada procedimiento: **cuándo se usa · pasos exactos con comandos · cómo verificar que funcionó · cómo se revierte**:

1. Deploy y rollback por app (incluye el caso "rollback de API con cambio de contrato": redeployar FEs compatibles).
2. Restore de backup (link a F5-07 + decisión de cuándo restaurar vs reparar).
3. Rotación de secretos: `JWT_SECRET` (efecto: access tokens mueren ≤15 min — momento recomendado), `RESEND_API_KEY`, connection string Atlas, clave `age` de backups.
4. Incidente de seguridad: los 5 pasos de [Seguridad §9](../05-seguridad.md) aterrizados con comandos reales (revocar familias: script `scripts/revoke-all-sessions.ts`; suspender org: update de `status`).
5. Soporte nivel 1 (las FAQs que dejará la beta): reenviar verificación · atleta no encuentra el código · admin bloqueado (reset por owner) · "no me deja cancelar" (explicar ventana) · cambiar email de una cuenta.
6. Salud diaria: qué mirar (Sentry, uptime, métricas Atlas) y umbrales de alarma.

### Casos de prueba
**Validación por ejecución**: una persona sin contexto del código (o una sesión limpia de un modelo, sin este repo en memoria) ejecuta **el rollback de una app** y **una rotación de `RESEND_API_KEY`** siguiendo solo el runbook. Donde se trabe → el doc está incompleto → se corrige.

### Criterios de aceptación
- [ ] Los 6 procedimientos escritos con comandos copy-paste.
- [ ] Validación por ejecución hecha para los 2 procedimientos indicados (notas de la corrida en el PR).
- [ ] `scripts/revoke-all-sessions.ts` existe y está testeado.
