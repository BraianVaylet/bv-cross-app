## F4-01 ⛔ — `booking-service`: reservar y cancelar (transaccional)

**Dificultad**: 🔴 Senior · **Modelo**: Fable · **Depende de**: F3-01, F3-03 · **Bloquea**: F4-02, y completa F3-01 (cancel de sesión con reservas)
**Docs**: **[Arquitectura §6 — implementar EXACTAMENTE ese algoritmo](../02-arquitectura.md)** · RN-06/07/08/09/10/11/12/13 · [Testing §4](../08-testing.md)

### Objetivo
Un servicio de dominio (`modules/bookings/booking-service.ts`) con tres operaciones transaccionales: `book`, `cancelByUser`, `cancelSessionByGym`. Acá viven los créditos de gente que pagó: la corrección bajo concurrencia no es negociable.

### Especificación técnica

**`book(userId, orgId, sessionId): Booking`** — transacción Mongo (`withTransaction`, `readConcern: 'snapshot'`, `writeConcern: 'majority'`), condiciones **dentro** de los updates (nunca check-then-act):

```
1. session = classSessions.findOneAndUpdate(
     { _id: sessionId, orgId, status: 'scheduled', startsAt: { $gt: now },
       $expr: { $lt: ['$bookedCount', '$capacity'] } },
     { $inc: { bookedCount: 1 } }, { returnDocument: 'after' })
   null → abort + DIAGNÓSTICO fuera de la tx (read simple para elegir el código):
          no existe/otra org → NOT_FOUND · cancelled → SESSION_CANCELLED
          startsAt <= now → SESSION_STARTED · lleno → SESSION_FULL

2. pack = packAssignments.findOneAndUpdate(
     { userId, orgId, status: 'active',
       startsAt: { $lte: now }, expiresAt: { $gt: now },          // RN-11 + inicio futuro no usable (F3-03)
       $expr: { $lt: ['$classesUsed', '$snapshot.classCount'] } },
     [ { $set: { classesUsed: { $add: ['$classesUsed', 1] },
                 status: { $cond: [ { $gte: [{ $add: ['$classesUsed', 1] }, '$snapshot.classCount'] },
                                    'exhausted', 'active' ] },      // transición RN-13 en el MISMO update
                 updatedAt: '$$NOW' } } ],
     { sort: { expiresAt: 1 }, returnDocument: 'after' })          // FIFO RN-12
   null → abort (rollback del inc de sesión) → NO_CREDITS

3. bookings.insertOne({ orgId, sessionId, userId, packAssignmentId: pack._id,
                        status: 'booked', bookedAt: now })
   E11000 (índice parcial RN-07) → abort → ALREADY_BOOKED
```
Reintento ante `TransientTransactionError` / `UnknownTransactionCommitResult`: hasta 3 veces con backoff corto (helper `withRetry` documentado).

**`cancelByUser(userId, orgId, bookingId): { refunded: true }`** — transacción:
1. `bookings.findOneAndUpdate({ _id, userId, orgId, status: 'booked' }, { $set: { status: 'cancelled_by_user', cancelledAt: now } })` → null → `NOT_FOUND`.
2. Leer sesión; **ventana RN-08**: `session.startsAt - now >= org.settings.cancellationWindowHours * 3600s`, si no → abort → `CANCELLATION_WINDOW_CLOSED` (con `details.deadline` = instante límite). Sesión ya cancelada por el gym → el booking ya estaría cancelado (ver abajo) → naturalmente `NOT_FOUND`.
3. `classSessions.updateOne({ _id, $expr: { $gt: ['$bookedCount', 0] } }, { $inc: { bookedCount: -1 } })`.
4. Devolución al pack de origen (`packAssignmentId` del booking — **no** al FIFO actual): `$inc classesUsed: -1` + pipeline: si `status === 'exhausted'` y `expiresAt > now` → vuelve a `'active'` (reparación RN-13). Si el pack está `expired`/`cancelled`: el crédito se decrementa igual pero el estado no cambia — crédito devuelto pero inutilizable (RN-08, documentado en código).

**`cancelSessionByGym(orgId, sessionId, actorUserId)`** (reemplaza el guard `HAS_BOOKINGS` de F3-01):
1. Marca `classSessions.status = 'cancelled'` (idempotente: ya cancelada → no-op).
2. Itera bookings `booked` de la sesión: por cada uno, transacción de devolución **sin validar ventana** (RN-09), `status: 'cancelled_by_gym'`. Falla parcial: reintenta por booking; si uno falla definitivamente → log error con bookingId y sigue (el job/retry manual lo resuelve; nunca dejar la sesión a medio cancelar silenciosamente — la operación reporta `{ cancelled, failed }`).

### Casos de prueba (integración contra replica set — [Testing §4](../08-testing.md) los 6 + bordes)
**Concurrencia (los 6 canónicos)**:
1. `capacity: 1`, 2 `book` en `Promise.all` → exactamente 1 OK y 1 `SESSION_FULL`; `bookedCount === 1`; 1 booking en DB.
2. Pack con 1 crédito, 2 sesiones distintas, 2 book simultáneos → 1 OK, 1 `NO_CREDITS`; `classesUsed === 1`.
3. Doble-tap: mismo user+sesión ×2 → 1 booking, el otro `ALREADY_BOOKED`; **contadores en 1** (el rollback de la tx no deja el `bookedCount` inflado).
4. Rollback: fallo inyectado en el paso 3 (mock del insert) → `bookedCount` y `classesUsed` intactos.
5. Cancel concurrente con book (última plaza): estados finales consistentes — nunca `bookedCount` negativo ni sobrecupo (correr 20 iteraciones).
6. Stress: 20 books concurrentes a `capacity: 10` → 10 OK / 10 `SESSION_FULL`, contadores exactos.
**FIFO y estados**:
7. Packs A (vence en 5 días) y B (en 20) → book consume A; A agotado → siguiente book consume B (RN-12).
8. Último crédito → pack pasa a `exhausted` **en el mismo update** (leer inmediatamente después: nunca se observa `active` con `classesUsed == classCount`).
9. Refund a `exhausted` no vencido → vuelve `active`; a `expired` → sigue `expired` con crédito decrementado.
10. Pack con `startsAt` mañana → `NO_CREDITS` hoy (no usable antes del inicio).
**Ventana (RN-08)**:
11. Ventana 2 h: cancelar a 2h01 → OK; a 1h59 → `CANCELLATION_WINDOW_CLOSED`; **exactamente 2h00** → OK (regla: `>=`, fijada acá).
12. Ventana 0 → cancelable hasta `startsAt`.
**Gym cancel (RN-09)**: 13. sesión con 3 reservas (una dentro de la ventana cerrada) → las 3 devueltas, packs correctos, session `cancelled`; re-invocación → no-op idempotente.
**Reserva inválida**: 14. sesión pasada → `SESSION_STARTED` · cancelada → `SESSION_CANCELLED` · de otra org → `NOT_FOUND`.

### Criterios de aceptación
- [ ] Los 14 casos verdes; cobertura ≥ 90% del módulo (umbral en CI — [Testing §7](../08-testing.md)).
- [ ] Cero `find` + `update` separados para decisiones de cupo/crédito (revisión de código: toda condición vive en el filtro del update).
- [ ] El guard temporal `HAS_BOOKINGS` de F3-01 eliminado y reemplazado.
- [ ] Códigos de error nuevos (`SESSION_CANCELLED`, etc.) ya estaban en contracts F0-02 — verificar mapeos.

## F4-02 ⛔ — Endpoints de bookings + saldo

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F4-01
**Docs**: [Arquitectura §9](../02-arquitectura.md) · RN-07/08

### Especificación técnica

| Método y ruta | Guard | Descripción |
|---|---|---|
| `POST /bookings` | member | `{ sessionId }` → 201 `{ booking, session: { id, bookedCount, capacity }, credits: { remaining, packName, expiresAt } }` (el FE actualiza sin refetch) |
| `POST /bookings/:id/cancel` | member | 200 `{ refunded: boolean, credits: {...} }` |
| `GET /me/bookings?scope=upcoming\|history&after=&limit=` | member | Con sesión embebida (`$lookup`): fecha/hora/disciplina. `upcoming`: `booked` con `startsAt > now` asc; `history`: resto desc |
| `GET /me/credits` | member | `{ packs: [{ id, name, remaining, total, expiresAt, status, startsAt }], totalRemaining, nextExpiration }` — activos orden `expiresAt asc` (el primero es el que consumirá FIFO) |

- Los POST son envoltorios finos del servicio F4-01 (parse → servicio → DTO); cero lógica propia.
- `GET /me/credits` incluye packs `active` con `startsAt` futuro marcados (`usableFrom`) — el FE los muestra "disponible desde el lunes".

### Casos de prueba
1. Integración de cada endpoint feliz + mapeo de TODOS los códigos de error del servicio (tabla → assert por código).
2. `POST /bookings` responde `credits.remaining` correcto tras el consumo (coherencia respuesta/DB).
3. `upcoming`/`history` con fixture mixta: particiones correctas y órdenes correctos; paginación estable.
4. `GET /me/credits`: 3 packs (activo, activo futuro, agotado) → orden y campos correctos; `totalRemaining` suma solo usables ahora.
5. Suite F1-09: políticas + factory `booking`; atleta A no cancela bookings de B (404).

### Criterios de aceptación
- [ ] Contracts completos (`bookingDto`, `creditsDto`, bodies) — son la interfaz de F4-04/05/06.
- [ ] 5 grupos de casos verdes.
- [ ] Seed extendido: reservas de ejemplo (F1-11).

## F4-04 — Grilla semanal + flujo de reserva

**Dificultad**: 🔴 Senior · **Modelo**: Opus · **Depende de**: F4-02, F4-03
**Docs**: [Funcional F4, §8 (metas UX)](../01-funcional.md) · [Design System §4](../04-design-system.md) · [Arquitectura §7 regla 4](../02-arquitectura.md)

### Objetivo
La pantalla más usada del producto: ver la semana y reservar en menos de 10 segundos y 3 taps.

### Especificación técnica
- **`WeekGrid`** (nuevo en `@bv/ui`): tabs de día (Lun 7 · Mar 8 · …) con hoy preseleccionado, swipe/flechas para semana siguiente/anterior (límite: no navegar más allá del horizonte materializado — la API devuelve vacío y la UI lo explica: "Las reservas se abren N días antes").
- **`SessionCard`** (nuevo en `@bv/ui`): hora (tz de la org — nunca la del dispositivo), disciplina, descripción corta, ocupación (`7/12` + barra; `warn` ≥ 80%), estado visual: disponible · casi llena · **llena** (deshabilitada, "Completa") · **reservada** (check + borde `accent`) · cancelada (tachada) · pasada (atenuada).
- **`CreditBadge`** (nuevo en `@bv/ui`): fijo en el header: "5 clases · vence 01/08" (datos de `GET /me/credits`, el primero FIFO). Cero créditos → variante `warn` "Sin clases — hablá con tu gimnasio". Tap → `/credits`.
- **Flujo de reserva**: tap en sesión disponible → sheet de confirmación: sesión (día/hora/disciplina) + origen del crédito ("Se descuenta de: 8 clases — te quedan 4") + CTA "Reservar" → POST → éxito: card pasa a reservada + `CreditBadge` actualizado **con la respuesta del POST** (sin refetch) + toast.
- **Errores con recuperación**: `SESSION_FULL` → toast "Se llenó justo ahora 😞" + refetch del día (la card se actualiza) · `NO_CREDITS` → sheet con CTA a `/credits` · `ALREADY_BOOKED` → estado reservada (idempotencia visual) · red caída → retry manual.
- Optimismo **acotado**: la card muestra spinner durante el POST; no se marca reservada hasta el 201 (con dinero/créditos: pesimista > optimista; decisión documentada).
- Estados: skeleton de grilla, día vacío ("No hay clases este día"), error de carga con retry.

### Casos de prueba
1. Unit: agrupado de sesiones por día en tz org (incluye el caso 23:30 AR / 02:30Z — mismo helper compartido que F3-06, importado de un único lugar).
2. Unit: navegación de semanas respeta el horizonte (mock de settings).
3. Component `SessionCard`: los 6 estados renderizan diferenciados (snapshot ligero por estado); llena y pasada no disparan `onSelect`.
4. Component flujo: confirmación muestra el pack FIFO correcto; 201 actualiza card+badge sin llamada extra (spy de fetch: exactamente 1 POST, 0 GET).
5. Component errores: `SESSION_FULL` dispara refetch del día (spy); `NO_CREDITS` navega a créditos.
6. Manual cronometrado: **app abierta → reserva confirmada < 10 s, ≤ 3 taps** (métrica Funcional §8; video en el PR).

### Criterios de aceptación
- [ ] Métrica de 3 taps verificada y grabada.
- [ ] Los 3 componentes nuevos en `@bv/ui` con tests.
- [ ] Horas correctas con dispositivo en otra timezone (test manual cambiando la tz del sistema, documentado).
- [ ] Todos los códigos de error de F4-02 tienen tratamiento visible (tabla código→UX en el PR).

## F4-05 — Mis reservas + cancelación

**Dificultad**: 🟡 Semi-senior · **Modelo**: Sonnet · **Depende de**: F4-02, F4-03
**Docs**: RN-08 · [Funcional F4](../01-funcional.md)

### Especificación técnica
- `/bookings`, tabs **Próximas** / **Historial** (paginado "cargar más").
- Card de reserva próxima: día/hora/disciplina + **estado de la ventana**: "Podés cancelar hasta las 16:00" (deadline calculado client-side con `settings.cancellationWindowHours` — la org viaja en `orgDto`) o "Ya no se puede cancelar" con el botón deshabilitado (**la regla se comunica antes del error**; el server igual re-valida).
- Cancelar → `ConfirmDialog`: "Se te devuelve 1 clase al pack {nombre}" (o advertencia si el pack de origen está vencido: "El crédito vuelve a un pack vencido: no podrás usarlo" — dato derivable de `GET /me/credits` + `packAssignmentId` del booking) → POST → toast + actualizar lista y badge.
- "Cambiar de horario" (editar F4 del Funcional) = cancelar (mismo flujo) + deep-link a la grilla en el día de la sesión cancelada.
- Historial: fecha, estado (`Badge`: asistida*/cancelada por vos/cancelada por el gym — *cuando exista asistencia).
- Cross-check F3-11: cambiar la ventana en el CRM → el deadline mostrado cambia (mismo dato de la org).

### Casos de prueba
1. Unit `cancellationDeadline(session, settings, tz)`: ventana 2 h, sesión 18:00 AR → deadline 16:00 AR (assert en UTC); ventana 0 → deadline = startsAt.
2. Component: dentro de ventana → botón activo con hora límite; fuera → deshabilitado con explicación; borde exacto (deadline == now) → activo (regla `>=` de F4-01, espejada).
3. Component: `CANCELLATION_WINDOW_CLOSED` del server (carrera reloj cliente/server) → mensaje claro + refresh del estado, sin crash.
4. Component: advertencia de pack vencido aparece solo cuando corresponde.
5. Manual: cancelar y re-reservar otro horario en < 20 s.

### Criterios de aceptación
- [ ] Countdown/deadline consistente con el server (misma regla, testeada en ambos lados).
- [ ] Flujo de edición completo.
- [ ] Estados vacíos de ambos tabs.

## F4-08 — E2E Playwright de los 4 flujos críticos

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F2-07, F3-12, F4-07 (apps completas local)
**Docs**: [Testing §2, §7](../08-testing.md)

### Especificación técnica
- Paquete `e2e/` en la raíz (workspace propio, no se deploya): Playwright + config multi-app (baseURLs de las 3 apps locales + API con seed fresco por corrida: `globalSetup` ejecuta `db:seed`).
- Selectores por `data-testid` (agregar a los componentes involucrados — permitido tocar apps para testeabilidad).
- **Flujos** (1 spec cada uno):
  1. `signup-join`: registro → verificación (leer link del email consola vía endpoint dev-only `GET /__test__/last-email` **solo `NODE_ENV=test`**, o parsear stdout del server — elegir y documentar) → join con código → Home con catálogo.
  2. `booking`: login atleta seed → reservar en la grilla → badge decrementa → cancelar desde Mis reservas → badge restaurado.
  3. `crm-assign`: login owner → alta de cliente → asignar pack con pago → visible en la ficha.
  4. `loads`: login atleta → registrar RM en ejercicio del catálogo → login owner en CRM → visible en Progreso del atleta.
- Config anti-flake: `retries: 1`, `trace: 'on-first-retry'`, screenshots/video on-failure, workers: 1 (los flujos comparten seed).
- CI: workflow `e2e.yml` aparte — nightly + manual (`workflow_dispatch`), no bloquea PRs ([Testing §7](../08-testing.md)).

### Casos de prueba
Los 4 specs son los casos. Meta-verificación: **5 corridas consecutivas verdes** (local o CI) sin flakes — requisito de aceptación explícito.

### Criterios de aceptación
- [ ] 4 specs verdes ×5 corridas (log adjunto).
- [ ] Workflow nightly configurado con artefactos (trace/video) en fallos.
- [ ] README de `e2e/`: cómo correr local, cómo leer un trace.
- [ ] El endpoint `__test__` (si se eligió) inaccesible fuera de `NODE_ENV=test` (test negativo incluido).
