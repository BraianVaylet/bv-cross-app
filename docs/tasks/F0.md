## F0-01 ⛔ — Inicializar el monorepo + `@bv/config`

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: — · **Bloquea**: todo
**Docs**: [Técnico §3, §7](../03-tecnico.md) · [Seguridad §5](../05-seguridad.md) · DEC-06

### Objetivo
Que el repo quede operativo como workspace pnpm + Turborepo, con la configuración de lint/format/TS centralizada que usarán todos los paquetes.

### Alcance
**Incluye**: workspace, turbo, `.npmrc`, `.gitignore`, estructura de carpetas vacía, paquete `@bv/config` completo, paquete de ejemplo para validar.
**No incluye**: CI (F0-04), código de apps, `@bv/contracts`/`@bv/ui`.

### Especificación técnica

Archivos a crear en la raíz:

```yaml
# pnpm-workspace.yaml
packages:
  - "apps/*"
  - "packages/*"
```

```jsonc
// package.json (raíz) — solo orquestación, sin dependencias de runtime
{
  "name": "bv-cross",
  "private": true,
  "packageManager": "pnpm@10.x",            // versión exacta instalada
  "engines": { "node": ">=22" },
  "scripts": {
    "dev": "turbo run dev",
    "build": "turbo run build",
    "lint": "turbo run lint",
    "typecheck": "turbo run typecheck",
    "test": "turbo run test",
    "format": "prettier --write .",
    "audit:deps": "pnpm audit --prod --audit-level=high"
  },
  "devDependencies": { "turbo": "^2", "prettier": "^3" }
}
```

```jsonc
// turbo.json — pipelines con grafo: build de un paquete depende del build de sus deps
{
  "tasks": {
    "build":    { "dependsOn": ["^build"], "outputs": ["dist/**"] },
    "lint":     { "dependsOn": ["^build"] },
    "typecheck":{ "dependsOn": ["^build"] },
    "test":     { "dependsOn": ["^build"] },
    "dev":      { "cache": false, "persistent": true }
  }
}
```

```ini
# .npmrc
minimum-release-age=4320        # 3 días — mitiga supply-chain de paquetes recién publicados
engine-strict=true
```

Estructura: `apps/{api,cross,schedule,crm}/.gitkeep`, `packages/{ui,contracts,config}/`.

`packages/config` (`@bv/config`), exports:

| Export | Contenido |
|---|---|
| `@bv/config/eslint/base` | flat config: `typescript-eslint` preset `strictTypeChecked` + reglas: `@typescript-eslint/no-floating-promises: error`, `no-explicit-any: error`, `consistent-type-imports: error`, `import/no-cycle: error` |
| `@bv/config/eslint/react` | base + `eslint-plugin-react-hooks` (`rules-of-hooks: error`, `exhaustive-deps: warn`) |
| `@bv/config/tsconfig.base.json` | `strict: true`, `module/moduleResolution: NodeNext` (API) — variante `tsconfig.vite.json` con `bundler` para FEs, `target: ES2022`, `noUncheckedIndexedAccess: true`, `verbatimModuleSyntax: true` |
| `@bv/config/prettier` | `printWidth: 100`, `singleQuote: true`, `trailingComma: 'all'` |

Cada workspace futuro consume: `"@bv/config": "workspace:*"` + `eslint.config.js` de 3 líneas que importa el preset.

### Casos de prueba
Crear `packages/config/__smoke__/` (o usar `packages/contracts` como conejillo cuando exista):
1. Archivo con promesa flotante (`void` omitido) → `turbo run lint` **falla** con `no-floating-promises`.
2. Archivo con `any` explícito → lint falla.
3. Archivo mal formateado → `prettier --check` falla; `format` lo arregla.
4. `tsconfig` extendido detecta acceso a índice sin check (`noUncheckedIndexedAccess`).
5. Segunda corrida de `turbo run lint` sin cambios → **cache hit** (`FULL TURBO` en output).

### Criterios de aceptación
- [ ] `pnpm install` desde cero funciona en máquina limpia (documentado en README raíz: requisitos Node 22 + pnpm).
- [ ] `turbo run lint typecheck` verde sobre el paquete de ejemplo.
- [ ] Los 5 casos de prueba de arriba verificados (los negativos fallan, el cache pega).
- [ ] `.npmrc` con `minimum-release-age` activo (verificar: `pnpm config get minimum-release-age`).
- [ ] Ninguna dependencia de runtime en la raíz (solo `turbo` y `prettier` como dev).

### Sí o sí (no negociable)
- `workspace:*` como único protocolo interno; prohibido `file:` o versiones fijas internas.
- El lockfile `pnpm-lock.yaml` se commitea en este PR y en todos los siguientes.

## F0-02 ⛔ — `@bv/contracts`: núcleo

**Dificultad**: 🟡 Semi-senior · **Modelo**: Opus · **Depende de**: F0-01 · **Bloquea**: F1-01+, todo FE
**Docs**: [Técnico §4](../03-tecnico.md) · [Arquitectura §4](../02-arquitectura.md) · [Seguridad §2](../05-seguridad.md)

### Objetivo
Una única fuente de verdad para tipos y validación entre API y frontends: enums del dominio, primitivas, envelope de error, catálogo de códigos y matriz de permisos.

### Alcance
**Incluye**: primitivas, enums, errores, permisos, paginación. **No incluye**: DTOs por recurso (se agregan en la tarea del módulo correspondiente — cada módulo de API suma sus schemas acá en su propio PR).

### Especificación técnica

Estructura `packages/contracts/src/`:

```
core.ts        # primitivas: objectIdString, isoDateString, calendarDate, email, cursor
errors.ts      # ErrorCode (const), errorEnvelope schema, HTTP status por código
enums.ts       # enums del dominio (fuente única)
permissions.ts # matriz rol → acciones
pagination.ts  # cursorQuery, page<T> helper
index.ts       # barrel
```

Definiciones exactas:

```ts
// core.ts
export const objectIdString = z.string().regex(/^[0-9a-f]{24}$/i);
export const isoDateString  = z.string().datetime({ offset: false }); // siempre UTC 'Z'
export const calendarDate   = z.string().regex(/^\d{4}-\d{2}-\d{2}$/); // fechas sin hora (rmEntries)
export const email          = z.string().trim().toLowerCase().email().max(254);

// enums.ts — valores EXACTOS de Arquitectura §4 (cambiarlos = breaking, requiere migración)
export const role                 = z.enum(['owner', 'admin', 'coach', 'athlete']);
export const membershipStatus     = z.enum(['invited', 'active', 'disabled']);
export const exerciseType         = z.enum(['weight', 'reps']);
export const exerciseScope        = z.enum(['org', 'personal']);
export const packAssignmentStatus = z.enum(['active', 'exhausted', 'expired', 'cancelled']);
export const bookingStatus        = z.enum(['booked', 'cancelled_by_user', 'cancelled_by_gym', 'attended', 'no_show']);
export const paymentMethod        = z.enum(['cash', 'debit', 'transfer', 'other']);
export const sessionStatus        = z.enum(['scheduled', 'cancelled']);
```

Catálogo inicial de `ErrorCode` (extensible, nunca se renombra uno publicado):
`VALIDATION_ERROR(400) · WEAK_PASSWORD(400) · ORG_HEADER_MISSING(400) · INVALID_CREDENTIALS(401) · TOKEN_EXPIRED(401) · TOKEN_INVALID(401) · EMAIL_NOT_VERIFIED(403) · NOT_A_MEMBER(403) · FORBIDDEN_ROLE(403) · BAD_ORIGIN(403) · NOT_FOUND(404) · ORG_CODE_INVALID(404) · EMAIL_TAKEN(409) · ALREADY_MEMBER(409) · SESSION_FULL(409) · SESSION_CANCELLED(409) · SESSION_STARTED(409) · NO_CREDITS(409) · ALREADY_BOOKED(409) · CANCELLATION_WINDOW_CLOSED(409) · PACK_IN_USE(409) · TYPE_LOCKED(409) · LAST_ENTRY(409) · CAPACITY_BELOW_BOOKED(409) · MEMBER_DISABLED(409) · CANNOT_MODIFY_OWNER(403) · RATE_LIMITED(429) · INTERNAL(500)`

Matriz de permisos (RN-04) como const tipada:

```ts
export const PERMISSIONS = {
  'org:settings': ['owner'],
  'org:regenerate-code': ['owner'],
  'org:manage-admins': ['owner'],
  'members:manage': ['owner', 'admin'],
  'schedule:manage': ['owner', 'admin'],
  'packs:manage': ['owner', 'admin'],
  'assignments:manage': ['owner', 'admin'],
  'exercises:manage-catalog': ['owner', 'admin'],
  'stats:read': ['owner', 'admin'],
} as const satisfies Record<string, readonly Role[]>;
export const can = (role: Role, action: keyof typeof PERMISSIONS) => ...
```

Reglas: todo schema de objeto usa `.strict()` · tipos solo por `z.infer` · sin dependencias más que `zod`.

### Casos de prueba
1. `objectIdString` acepta 24 hex, rechaza 23 chars, mayúsculas mezcladas OK, rechaza `null`/número.
2. `email` normaliza (`" Foo@BAR.com "` → `"foo@bar.com"`), rechaza sin TLD y >254 chars.
3. `calendarDate` rechaza `2026-13-01`? (regex no valida rangos → agregar `.refine` con `Date` parse y test `2026-02-30` rechazado).
4. Todo schema `.strict()`: objeto con campo extra → rechazado (test genérico que itera los exports).
5. `can('athlete', 'members:manage')` → false; `can('owner', *)` → true para toda acción.
6. Snapshot de enums: test que fija los valores literales (si alguien cambia un valor, el test lo hace explícito).

### Criterios de aceptación
- [ ] `pnpm --filter @bv/contracts build` emite `dist/` con `.d.ts`.
- [ ] `exports` de package.json correctos (ESM only) — consumible desde Node y Vite.
- [ ] 100% de los schemas con test (es la base de todo: acá la cobertura sí es total).
- [ ] Documentado en el README del paquete: "cómo agregar un DTO nuevo" (convención de nombres `xxxBody`, `xxxDto`, `xxxQuery`).

- [ ] React como `peerDependency` (una sola copia de React en cada app).
- [ ] Cero valores de color crudos en los componentes (grep de `#[0-9a-f]` en `components/` = solo Icons).
- [ ] Los 7 grupos de casos de prueba verdes.

## F0-04 — CI del monorepo

**Dificultad**: 🟢 Junior · **Modelo**: Sonnet · **Depende de**: F0-01 · **Bloquea**: F0-05
**Docs**: [Testing §7](../08-testing.md) · [Seguridad §5](../05-seguridad.md)

### Objetivo
Un workflow que garantice que nada entra a `main` sin lint, tipos, tests y auditoría de dependencias.

### Especificación técnica

`.github/workflows/ci.yml`:

```yaml
name: CI
on:
  pull_request:
  push: { branches: [main] }
concurrency: { group: ci-${{ github.ref }}, cancel-in-progress: true }
jobs:
  ci:
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: pnpm }
      - uses: actions/cache@v4            # cache de turbo
        with: { path: .turbo, key: turbo-${{ github.sha }}, restore-keys: turbo- }
      - run: pnpm install --frozen-lockfile
      - run: pnpm turbo run lint typecheck test build
      - run: pnpm audit --prod --audit-level=high
      - uses: google/osv-scanner-action@v2   # o binario directo
        with: { scan-args: --lockfile=pnpm-lock.yaml }
```

Notas: `--frozen-lockfile` obligatorio (lockfile desactualizado = build rojo) · versiones de actions pinneadas por major · el job entero < 5 min (con cache).

### Casos de prueba
1. PR con error de lint → rojo en el paso turbo.
2. PR que agrega dependencia sin actualizar lockfile → rojo en install.
3. PR limpio → verde en < 5 min; segunda corrida con cache más rápida.
4. Simulación de CVE: agregar temporalmente (en branch de prueba, no mergear) una versión vieja vulnerable conocida → `pnpm audit` rojo. Documentar la corrida en el PR.

### Criterios de aceptación
- [ ] Los 4 casos de prueba ejecutados y evidenciados (links a corridas en el PR).
- [ ] `concurrency` cancela corridas obsoletas del mismo PR.
- [ ] Badge de CI en el README raíz.

## F0-06 — Decidir y comprar dominio

**Dificultad**: 🟢 Junior · **Modelo**: humano (decisión de marca) · **Depende de**: — · **Bloquea**: F6-01
**Docs**: [Deployment §1](../06-deployment.md) · [Arquitectura §5](../02-arquitectura.md)

### Especificación
- Requisitos del dominio: un **apex único** para todo el producto (restricción SSO); corto y pronunciable en español; TLD confiable (`.app` fuerza HTTPS vía HSTS preload — recomendado; alternativas `.fit`, `.com.ar`, `.io`).
- Verificar disponibilidad de los subdominios previstos: `api.` `app.` `agenda.` `crm.`.
- Comprar en registrador con API DNS decente (Cloudflare Registrar recomendado: precio de costo + DNS rápido).
- **No configurar DNS todavía** (eso es F6-01) — solo asegurar la propiedad.

### Criterios de aceptación
- [ ] Dominio comprado, auto-renovación activada, 2FA en la cuenta del registrador.
- [ ] Decisión y dominio anotados en `docs/06-deployment.md` (reemplazar `<apex>` en los ejemplos).
